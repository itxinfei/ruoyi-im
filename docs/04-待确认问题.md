## 六、处理规则 (Anti-Hallucination Defense)

| 规则 | 说明 |
|-----|------|
| **未确认前** | 不得在需求文档、架构文档或代码中写成既定事实。 |
| **模型遇到未知** | **触碰红线法则即刻停机（HALT）**。若所需接口、状态、结构缺失，大模型严禁通过“常识”或“它库经验”进行凭空脑补（如伪造字段、捏造类名）。必须立即输出告警并请求用户输入。 |
| **确认后** | 同步更新需求文档 `12-需求点清单.md`、任务池、实现状态矩阵，确保 SSOT (Single Source of Truth) 不被稀释。 |
| **优先级定义** | P0 阻塞开发，P1 影响验收，P2 可后续迭代。 |
| **定期清理机制** | 强制执行清理机制，绝不允许悬挂未决问题超过 7 天。 |

## 七、定期清理机制

### 清理周期

| 周期 | 执行内容 | 责任人 |
|-----|---------|-------|
| **每日** | AI开发过程中发现新问题，即时登记 | 执行开发的AI |
| **每周** | 审查P0/P1待确认项，推动确认 | 项目负责人 |
| **每迭代** | 评审P2待确认项，决定是否处理 | 产品/研发 |

### 清理检查清单

```markdown
### 周清理检查清单

- [ ] P0待确认项是否全部处理？
- [ ] P1待确认项是否有积压超过3天？

- [ ] 已确认项是否同步到需求文档？

```

### 自动化提醒

当出现以下情况时，应在文档中添加提醒：

1. **P0积压**：P0待确认项超过0项，标记 `⚠️ 阻塞开发`
2. **P1积压**：P1待确认项超过5项，标记 `⚠️ 影响验收`
3. **长期未处理**：任何待确认项超过7天未处理

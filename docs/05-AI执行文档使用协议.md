# 05 AI执行文档使用协议

> **用途**：定义整套 `docs/` 作为大模型执行知识库时的读取顺序、优先级、停机条件、回写要求与交付格式。  
> **适用对象**：所有基于本项目文档进行规划、设计、开发、审查、优化的 AI / 大模型 / Agent。  
> **优先级**：仅次于用户当前指令；在文档体系内部，本文件负责“怎么用文档”，`02-大模型研发规范.md` 负责“不能越过什么红线”。

---

## 1. 定位

1. `docs/` 不是展示型资料库，而是 **AI 执行时的运行时知识库**。
2. 大模型不能只靠单篇文档或局部上下文工作，必须先按本文件完成文档路由，再开始设计、编码、审查或回写。
3. 如果文档之间存在冲突，必须按本文件定义的优先级处理；不能靠经验脑补。

---

## 2. 文档优先级

### 2.1 总体优先级顺序

| 优先级 | 文档层 | 代表文档 | 用途 |
|-----|------|---------|------|
| P0 | 当前任务指令 | 用户当前消息、明确 task_id、明确目标 | 当前任务唯一入口 |
| P1 | 运行协议与红线 | `05`、`02` | 决定怎么读文档、何时 HALT、哪些行为绝对禁止 |
| P2 | 产品与业务真相源 | `10`、`12`、`13`、`18`、`19` | 决定功能边界、业务规则、状态机与验收来源 |
| P3 | 页面与设计约束 | `15`、`16`、`20`、`21`、`22`、`25`、`26`、`27`、`35`、`49` | 决定 PC 版钉钉 UI 的页面模板、视觉规范、交互与 Token |
| P4 | 技术契约 | `14`、`17`、`30`、`31`、`32`、`33`、`34`、`36` | 决定路由、入口、API、数据库、WebSocket、架构约束 |
| P5 | 执行追踪与验收 | `03`、`04`、`11`、`23`、`24`、`52`、`54` | 决定任务状态、缺口、测试策略与人工验收 |
| P6 | 辅助模板与提示 | `LLM任务模板.md`、`llm-tasks.yaml`、`Claude提示词库.md` | 决定任务卡结构和提示词写法 |

### 2.2 冲突处理规则

1. **用户当前指令** 与文档冲突时，先执行用户指令；若存在明显风险，再指出冲突。
2. **`02-大模型研发规范.md`** 与其他任意文档冲突时，以 `02` 为红线基准。
3. **业务规则** 冲突时，优先看 `19` 与 `12` 中的 C 系列规则，再回看 `10/13/18`。
4. **UI 规则** 冲突时，先看 `27` 定页面类型，再看 `20/25` 定总规则与验收，再看 `21/26/49/35` 定细节。
5. 同一层级文档若互相矛盾且无法判断，**立即 HALT**，并把问题登记到 `04-待确认问题.md`。

## 3. 任务路由

### 3.1 按任务类型选择必读文档

| 任务类型 | 最小必读链路 |
|---------|-------------|
| UI 设计 / 钉钉复刻 / 前端交互优化 | `05 -> 01 -> 02 -> 27 -> 20 -> 25 -> 21/26/35/49 -> 14/17` |
| 后端功能 / 接口开发 / 状态机实现 | `05 -> 01 -> 02 -> 30 -> 13/18 -> 31/32/33 -> 34(如涉及WS)` |
| 全栈功能 / 复杂缺陷修复 | `05 -> 01 -> 02 -> 13/18 -> 14/17 -> 30/31/32/33/34 -> 20/25/27(如涉及UI)` |
| 路由/导航/入口改造 | `05 -> 01 -> 02 -> 14 -> 15 -> 17 -> 27/20/25(如涉及UI)` |
| 测试/验收/质量门禁 | `05 -> 02 -> 19 -> 52 -> 54 -> 03/24` |
| 文档优化 / 任务卡生成 / 提示词维护 | `05 -> 00 -> README -> LLM任务模板 -> Claude提示词库 -> 相关目标文档` |

### 3.2 UI 任务的固定路线

1. 先用 `27` 判断页面属于哪种 **PC 页面模板**。
2. 再用 `20` 锁定跨页面红线。
3. 再用 `25` 锁定页面级 Done 与验收等级。
4. 最后才进入 `21`、`26`、`35`、`49` 这些实现型或模块型规范。

### 3.3 后端任务的固定路线

1. 先用 `13` 或 `18` 锁定业务目标。
2. 再用 `31/32/33/34` 锁定实体、表、接口、信令。
3. 再用 `30` 确认实现方式、异常处理、日志、生命周期与约束。

## 4. AI 执行最小闭环

1. **识别任务类型**：先判断是 UI、后端、全栈、测试、文档还是复刻任务。
2. **完成文档路由**：按第 3 节读取最小必读链路，不允许跳过。
3. **执行存量扫描**：搜索现有代码、路由、接口、组件、样式、任务卡，禁止重复造轮子。
4. **锁定修改范围**：明确 `scope_in` / `scope_out`，避免大面积越界修改。
5. **实施与验证**：按文档约束完成实现、构建、检查和人工验证。
6. **同步文档回写**：代码变了，相关 docs 必须同步更新。
7. **输出结果**：明确改了什么、验证了什么、还有什么风险。

---

## 5. HALT 停机条件

满足任一项时，AI 必须停止脑补，向用户指出缺失点：

1. 接口、字段、事件、路由、组件名在代码或文档中都找不到。
2. 两份以上文档给出互相冲突的业务规则、状态机或验收标准。
3. 任务需要改动的入口、页面、模块边界不明确。
4. UI 任务无法确认页面模板、交互入口或验收口径。
5. 任务要求与 `02-大模型研发规范.md` 的红线直接冲突。

HALT 后的动作：

1. 明确指出缺失的事实或冲突文档。
2. 给出最小化待确认问题。
3. 如适用，将问题登记到 `04-待确认问题.md`。

## 7. 输出协议

AI 完成任务时，输出至少应包含：

1. **改动范围**：改了哪些文件或模块。
2. **关键原因**：为什么改、依据了哪些文档。
3. **验证结果**：构建、检查、手工验证是否通过。
4. **文档回写**：同步更新了哪些 docs。
5. **残留风险**：还有哪些未验证项、边界项或假设。

如果任务被 HALT，输出至少应包含：

1. 缺失事实
2. 受影响范围
3. 需要用户补充的最小信息

---

## 8. 完成定义

1. 任务本身完成。
2. 文档路由正确，没有依赖失效文档或过期路径。
3. 验证结果明确，不用“理论上可用”代替检查。
4. 该回写的文档已回写，索引与提示词未失联。

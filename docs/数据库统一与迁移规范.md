# 数据库统一与迁移规范（IM会话域）

## 1. 目标
- 以 `sql/im.sql` 作为唯一数据库真源。
- 解决会话域重复建模导致的数据不一致问题。
- 约束大模型开发行为，避免继续放大历史包袱。

## 2. 结论：三表职责重新定义
- `im_conversation`：
  - 会话主数据（会话类型、目标ID、最后消息ID/时间）。
- `im_conversation_member`：
  - 成员关系数据（谁在会话里、角色、群昵称、删除成员关系）。
  - 保留字段：`conversation_id/user_id/role/nickname/is_deleted/deleted_time/create_time/update_time`。
  - 逐步弃用字段：`unread_count/is_pinned/is_muted/is_archived/draft_content/draft_time/last_read_message_id/last_read_time`（迁移完成后不再作为读写真源）。
- `im_user_session`：
  - 用户会话状态真源（置顶、免打扰、归档、未读、已读游标、草稿、软删除）。
  - 状态类读写全部走此表。

## 3. 已落地改造（本次）
- 会话列表读链路改为“双表兼容读取，优先 `im_user_session`，回退 `im_conversation_member`”。
- 未读总数查询改为双表兼容聚合。
- 会话服务写链路已双写：
  - `pin/mute/clearUnread/markAsRead/deleteConversation`
  - 新表为主写（`im_user_session`），旧表兼容回写（`im_conversation_member`）
- 消息服务写链路已双写：
  - 发送消息增加他人未读数时同步更新 `im_user_session`
  - 已读回执推进时同步更新 `im_user_session` 的未读数与最后已读消息ID
- 补齐以下表的 `Mapper + XML`：
  - `im_user_session`
  - `im_typing_status`
  - `im_message_read_detail`

## 4. 后续分阶段迁移
- 阶段A（兼容期）：
  - 写路径保持现状，同时新增/补充对 `im_user_session` 的写入。
  - 读路径优先 `im_user_session`，不存在时回退 `im_conversation_member`。
- 阶段B（收敛期）：
  - 所有状态写入仅落 `im_user_session`。
  - `im_conversation_member` 仅用于成员关系。
- 阶段C（清理期）：
  - 下线 `im_conversation_member` 中状态字段的业务使用。
  - SQL 层考虑删除冗余字段（需评估历史数据导出/审计需求）。

## 5. 大模型开发强约束（必须遵守）
- 只允许基于 `sql/im.sql` 生成或修改实体、Mapper、XML、Service。
- 禁止重复创建同义表或同义实体（如 `*_setting` 与 `*_session` 重叠建模）。
- 禁止在未确认职责的情况下跨表写同一状态字段。
- 新增字段时必须同步更新：
  - Domain
  - Mapper 接口
  - Mapper XML：`resultMap`、查询列、`insert/update`
  - 至少一个编译验证命令
- 禁止“为了兼容”再新建一套平行接口/文件；优先在现有文件内增量改造。

## 6. 开发检查清单（每次提交前）
- 编译通过：`mvn -DskipTests compile`（`ruoyi-im-api`）
- 覆盖检查：
  - `@TableName` 对应表是否存在于 `sql/im.sql`
  - `Mapper.java` 与 `Mapper.xml` 是否成对存在
  - 状态字段是否只从 `im_user_session` 读取
- 回归点：
  - 会话列表排序（置顶优先、最近消息时间）
  - 未读数增减与清零
  - 草稿保存/清空
  - 已读游标推进

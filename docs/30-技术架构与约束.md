### 1.4 状态管理规范

```javascript
// stores/message.js
import { defineStore } from 'pinia';
import { messageApi } from '@/api/message';

export const useMessageStore = defineStore('message', {
  state: () => ({
    messagesMap: {},
    activeConversationId: null,
    loading: false,
  }),

  getters: {
    currentMessages: (state) => {
      if (!state.activeConversationId) return [];
      return state.messagesMap[state.activeConversationId] || [];
    },
  },

  actions: {
    async loadMessages(conversationId, lastMessageId) {
      this.loading = true;
      try {
        const { list } = await messageApi.getList({ conversationId, lastMessageId });
        if (!this.messagesMap[conversationId]) {
          this.messagesMap[conversationId] = [];
        }
        this.messagesMap[conversationId] = [...list, ...this.messagesMap[conversationId]];
      } finally {
        this.loading = false;
      }
    },

    pushMessage(conversationId, message) {
      if (!this.messagesMap[conversationId]) {
        this.messagesMap[conversationId] = [];
      }
      const exists = this.messagesMap[conversationId].some(m => m.id === message.id);
      if (!exists) {
        this.messagesMap[conversationId].push(message);
      }
    },
  },
});
```

### 1.5 内网部署约束 (Intranet Deployment)
1. **纯私有化运行**：所有服务资源必须支持100%离线局域网运行，禁止任何硬编码公网 CDN 依赖。前端资源必须全部本地化打包。
2. **端口与网关**：前后端统一经过 Nginx/网关 层进行端口收敛隔离分析，禁止客户端直连后端源站 IP 及敏感内网端口。

### 1.6 全局数据契约 (Data Contract)
1. **命名风格**：
   - **前端/API (JSON)**：一律使用 `camelCase` (小驼峰)，如 `clientMsgId`。
   - **后端 (Entity/DB)**：一律使用 `snake_case` (下划线)，如 `client_msg_id`。
   - **DTO/VO**：后端 DTO/VO 属性必须与前端 JSON 字段名严格对齐。
2. **布尔值传输**：
   - 接口传输统一使用 `Integer` (0 或 1) 代表布尔值，禁止在 JSON 中传输原生 `true/false`。
3. **时间标准**：
   - 所有接口的时间戳传输统一使用 **Long 型毫秒值** (如 `1710921600000`)。
   - 禁止在 API 传输 `yyyy-MM-dd` 格式字符串，格式化显示由前端独立负责。
4. **空值处理**：
   - 集合类型字段 (List) 若无数据必须返回 `[]`，严禁返回 `null`。
   - 字符串字段若无数据必须返回 `""`，严禁返回 `null`。

### 1.7 全链路追踪 (End-to-End Traceability)

为了解决内网复杂环境下的排障难题，系统必须实现全链路日志追踪：
1. **TraceId 生成**：由前端在用户触发动作（如点击发送、加载列表）时生成唯一 `traceId` (UUID)。
2. **REST 请求传递**：所有 API 请求必须在 Header 中携带 `X-Trace-Id`。
3. **WebSocket 传递**：所有 WS 数据帧必须在根层级携带 `traceId` 字段。
4. **后端透传**：后端接收后，必须将其注入 `MDC` (Mapped Diagnostic Context)，确保该请求产生的每一行 Log 均包含此 `traceId`。
5. **异常返回**：API 报错时，`responseBody` 必须包含此 `traceId`，方便用户反馈时一键定位。

### 1.9 长效运行与内存安全 (Memory Safety)

IM 客户端具备"长年不关"的特性，AI 在编写 Vue 组件时必须强制遵守销毁契约：

1. **成对注销监听**：
   - 凡是在 `onMounted` 或组件内部注册的 WebSocket 事件 (`ws.on`)、全局事件 (`mitt.on`)、DOM 监听，必须在 `onUnmounted` 中显示注销。
2. **定时器清理**：
   - 所有用于心跳、倒计时、轮询的 `setTimeout` 和 `setInterval` 必须在组件销毁时执行 `clearTimeout/clearInterval`。
3. **Store 状态复位**：
   - 当用户退出登录或切换账号时，Pinia Store 必须提供并调用 `$reset()` 方法，防止敏感数据残留在内存中。
4. **长列表缓存释放**：
   - 消息列表在切换会话时，若非当前活跃会话，应考虑适度清理超出缓存阈值（如 >500 条）的消息对象，以维持平稳的内存曲线。

### 2.5 异常处理

// 业务异常
public class BusinessException extends RuntimeException {
    private final int code;

    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }
}

// 全局异常处理
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public AjaxResult processBusinessException(BusinessException e) {
        return AjaxResult.error(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public AjaxResult processException(Exception e) {
        log.error("系统异常", e);
        return AjaxResult.error(500, "系统繁忙");
    }
}

## 四、WebSocket 协议

### 4.1 连接地址

```
ws://{server}/ws/im
```

### 4.2 消息帧格式

```json
{
  "type": "MESSAGE",
  "action": "SEND",
  "requestId": "uuid",
  "data": {},
  "timestamp": 1710921600000
}
```

### 4.3 消息类型

| 类型 | 说明 | 动作 |
|-----|------|------|
| AUTH | 鉴权与心跳 | PING, PONG |
| MESSAGE | IM 消息 | SEND, RECEIVE, REVOKE, EDIT, READ_RECEIPT |
| EVENT | 状态事件 | TYPING, PRESENCE_CHANGE |
| CALL | 通话信令 | INVITE, ANSWER, REJECT, END |

### 4.4 连接维护

- **鉴权**：连接后 5s 内发送 AUTH 帧（带 Token）
- **心跳**：每 30s 发送 PING
- **重连**：指数退避，最大间隔 30s

### 4.5 在线状态管理

- **存储**：使用 Redis 维护用户在线状态
- **Key格式**：`im:user:online:{userId}`
- **过期时间**：心跳超时 60s 后自动下线
- **并发控制**：单会话并发上限 500 人

## 五、开发约束

### 5.1 禁止行为

1. 禁止把未实现功能伪装成已完成
2. 禁止只改文案不打通事件
3. 禁止无关大面积重构
4. 禁止用猜测补全未知事实
5. 禁止使用外部 CDN
6. 禁止引入与需求无关的依赖
7. 绝对禁止捕获吞掉异常，必须通过抛出 BusinessException 遵循在 `docs/backend/02-全局错误码与异常字典规范.md` 定义的规范结构。
8. 绝对禁止在日志打印聊天消息明文，必须遵循 `docs/backend/03-日志掩码与可观测性规约.md`。

### 5.2 验证要求

- 每次开发必须运行 `npm run build` 验证
- 必须确保主流程可闭环
- 未知项登记到 `04-待确认问题.md`
- 消息长列表前端渲染必须符合 `docs/frontend/` 目录下的虚拟滚动要求
- 后端 WebSocket 分发必须符合 `docs/backend/` 目录下的 EventBus 异步解耦要求

### 5.3 完成定义

- 用户能从真实入口触达
- 主流程可闭环，不依赖占位提示
- 异常场景有明确反馈
- 前后端状态一致
- 构建验证通过

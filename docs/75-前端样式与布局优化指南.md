## 2. 前端样式与布局优化最佳实践

### 2.1 设计令牌优先，禁止页面私有视觉体系

所有颜色、字号、圆角、间距、阴影、尺寸都应优先来自 `src/styles/design-tokens.scss`。页面组件可以组合 token，但不能另建一套隐式 token 体系。

**执行规则：**

- 颜色：优先 `var(--dt-*)`，禁止随意新增十六进制色值。
- 间距：使用 4/8/12/16/24/32 的栅格步进，不要出现 `13px`、`18px` 这类无来源值。
- 尺寸：会话栏、抽屉、头部等核心区域，优先复用现有布局 token。
- 阴影：默认弱阴影，只在弹层、悬浮条、菜单等高层级元素使用。

**推荐写法：**

```scss
.panel {
  background: var(--dt-bg-card);
  border: 1px solid var(--dt-border-light);
  border-radius: var(--dt-radius-lg);
  box-shadow: var(--dt-shadow-1);
}
```

**避免写法：**

```scss
.panel {
  background: #fff;
  border: 1px solid #e5e7eb;
  border-radius: 10px;
  box-shadow: 0 10px 30px rgba(0, 0, 0, 0.12);
}
```

### 2.2 页面布局必须先解决“骨架”，再处理“细节”

页面质量差，通常不是某个按钮不精致，而是骨架不稳定。应先固定以下结构原则：

- 一级布局：`L1 导航 + L2 列表/侧栏 + L3 主内容`。
- 主内容容器必须 `min-width: 0`，避免文本、列表、弹层把容器撑爆。
- 滚动原则：一个区域只保留一个主滚动容器，避免“双滚动条”与滚动同步问题。
- 侧栏/详情抽屉优先使用 token 宽度，不要在组件里反复硬编码。

**推荐骨架：**

```scss
.page-shell {
  display: flex;
  width: 100%;
  height: 100%;
  overflow: hidden;
}

.page-main {
  flex: 1;
  min-width: 0;
  min-height: 0;
  display: flex;
  flex-direction: column;
}

.page-scroll {
  flex: 1;
  min-height: 0;
  overflow: auto;
}
```

### 2.3 响应式适配应从“组件容器”出发，而不是只盯住视口

本项目目标形态是桌面端，但不能把“桌面端”误解为“固定像素宽度且不可退化”。响应式不等于移动端折叠，而是确保在窄窗口、低分辨率、系统缩放下页面仍可用。

**推荐策略：**

- 使用 `clamp()` / `minmax()` / `max()` / `min()` 处理侧栏、卡片、内容区宽度。
- 优先让组件根据父容器适配；需要时引入 `@container` 做容器查询。
- 详情抽屉、大列表面板、右侧边栏在中小窗口下应支持覆盖式或折叠式退化。
- 避免把核心流程建立在“宽屏下才看得到”的 hover 交互上。

**推荐写法：**

```scss
.session-panel {
  width: clamp(220px, 24vw, var(--dt-session-panel-width));
  min-width: 220px;
  max-width: 280px;
}

.content-grid {
  display: grid;
  grid-template-columns: minmax(0, 1fr) minmax(280px, 320px);
  gap: var(--dt-spacing-lg);
}
```

### 2.4 交互体验必须显式覆盖 6 类状态

任何页面和组件都必须明确以下状态，而不是只实现“成功态”：

1. `loading`
2. `empty`
3. `error`
4. `disabled`
5. `hover/focus`
6. `retry/recovery`

如果页面只在正常数据下工作，大模型后续继续生成代码时会倾向于复制这种“单状态实现”，导致体验脆弱。

### 2.5 可访问性不是可选项

桌面端项目同样需要基础可访问性约束：

- 所有交互按钮必须可键盘聚焦。
- `:focus-visible` 必须保留，不得仅依赖 hover。
- 不要用 CSS 改变视觉顺序但破坏 DOM 阅读顺序。
- 动画与过渡应尊重 `prefers-reduced-motion`。
- 图标按钮必须具备 `title`、`aria-label` 或明确文本语义。

### 2.6 性能优化应分成“首屏性能”和“更新性能”

**首屏性能：**

- 路由级或模块级拆包，避免一次性加载全部业务面板。
- 图片、预览图、离屏媒体优先使用原生懒加载。
- 控制初始 JS 体积，减少非必要依赖与大组件同步注入。

**更新性能：**

- 保持子组件 props 稳定，减少联动重渲染。
- 大列表优先考虑虚拟滚动或分页。
- 高频输入、搜索、WebSocket 更新必须防抖/节流。
- 谨慎抽象组件层级，避免列表中出现过多包裹组件。

## 3. 大模型开发中的前端界面设计特殊需求与技术规范

### 3.1 大模型生成前端代码时的特殊风险

与人工开发相比，大模型更容易出现以下问题：

- 用“看起来合理”的硬编码色值替代 token。
- 在不了解页面骨架的情况下直接局部补样式，导致层级继续失控。
- 用多个滚动容器、多个定位上下文堆出“视觉正确但结构脆弱”的页面。
- 只实现 happy path，漏掉空态、异常态、窄宽度退化。
- 在未核实现有组件/样式文件的情况下重复造轮子。

### 3.2 大模型前端输出的强制规范

1. **先搜现有组件，再新增实现**。  
   页面、抽屉、列表、输入区、对话框优先复用已有结构。

2. **样式必须先落 token 再落组件**。  
   若发现缺 token，应先补 token，再改页面；不要直接在业务组件里埋颜色。

3. **布局先定义容器，再定义内部块**。  
   必须先回答“谁是滚动容器、谁负责伸缩、谁负责最小宽度”。

4. **页面必须显式声明空/错/载入状态**。  
   训练语料中若缺失这些状态，大模型后续输出会继续缺失。

5. **交互入口必须具备闭环**。  
   任一按钮、图标、hover 工具条、快捷键入口都要验证“点下去后是否真的有路径和反馈”。

6. **响应式必须写退化策略**。  
   不能只写“支持响应式”，必须说明：窄窗口下哪些区域缩放、折叠、覆盖或隐藏。

7. **全局监听、定时器、WebSocket 订阅必须成对清理**。  
   否则页面切换后会出现内存泄漏和重复事件。

### 3.3 推荐的大模型前端输出模板

当 AI 处理页面优化任务时，建议按以下顺序组织实现：

1. 现状定位：当前组件、样式文件、状态入口。
2. 结构判断：主布局、滚动容器、层级关系。
3. token 对齐：复用哪些 `--dt-*` 变量。
4. 状态设计：loading/empty/error/interactive。
5. 响应式退化：窄窗口下的布局策略。
6. 性能处理：拆包、懒加载、防抖、稳定 props。
7. 验证方式：截图对比、键盘导航、性能面板、构建体积。

## 4. 当前项目中的真实问题与解决方案示例

以下问题均来自当前仓库的文档和代码现状，可作为后续页面优化的直接参考样板。

### 4.1 问题一：文档 token 命名与代码 token 命名存在漂移

**现状证据：**

- `docs/35-前端全局样式与主题配置.md` 中同时出现 `--dt-spacing-s/m/l`、`--dt-radius-m` 等命名。
- `ruoyi-im-web/src/styles/design-tokens.scss` 实际使用的是 `--dt-spacing-sm/md/lg`、`--dt-radius-md`。

**风险：**

- 大模型按文档抄写时容易生成不存在的变量名。
- 页面代码与文档规范逐渐分叉，后续维护成本升高。

**解决方案：**

- 统一以 `src/styles/design-tokens.scss` 为运行时真相源。
- 文档中所有 token 示例都使用实际存在的 `--dt-*` 名称。
- 若要新增 token，必须先改 `design-tokens.scss`，再回写规范文档。

### 4.2 问题二：消息页侧栏宽度硬编码，未复用布局 token

**现状证据：**

- `ChatSessionList.vue` 使用 `width: 280px;`
- `design-tokens.scss` 已定义 `--dt-session-panel-width: 250px;`

**风险：**

- 视觉基线与文档不一致。
- 后续多个页面复制该写法后会继续分叉。

**推荐改法：**

```scss
.session-panel-v4 {
  width: clamp(220px, 24vw, var(--dt-session-panel-width));
  min-width: 220px;
  max-width: 280px;
}
```

**收益：**

- 默认仍以钉钉基线宽度为准。
- 在窄窗口下具备可退让能力，不会立刻挤爆主内容区。

### 4.3 问题三：消息页存在双滚动容器

**现状证据：**

- `ChatWindow.vue` 中 `.message-area-scroller` 自身 `overflow-y: auto`
- `ChatMessageList.vue` 中 `.v6-message-viewport` 也 `overflow-y: auto`

**风险：**

- 滚动事件可能挂在错误元素上。
- 上拉加载更多、滚动定位恢复、键盘滚动体验都会变脆弱。
- 页面滚动性能和交互一致性下降。

**推荐改法：**

```scss
.message-area-scroller {
  flex: 1;
  min-height: 0;
  overflow: hidden;
}

.v6-message-viewport {
  height: 100%;
  overflow: auto;
}
```

**执行原则：**

- 外层负责占位和高度约束。
- 内层负责真实滚动与滚动事件监听。

### 4.4 问题四：右侧详情区缺少窄窗口退化策略

**现状证据：**

- `ChatWindow.vue` 中详情区固定 `320px`
- 主聊天区最小宽度 `480px`
- 页面整体仍以桌面宽度假定为主

**风险：**

- 在较窄桌面窗口、浏览器分屏或系统缩放下出现内容挤压。
- 主内容、输入区、详情区抢空间，界面稳定性下降。

**推荐策略：**

- 宽窗口：保持 push drawer。
- 中等窗口：切为覆盖式抽屉。
- 更窄窗口：默认收起详情区，通过按钮显式打开。

**示例：**

```scss
@media (max-width: 1280px) {
  .chat-right-sidebar {
    position: absolute;
    top: 0;
    right: 0;
    bottom: 0;
    z-index: var(--dt-z-drawer);
    box-shadow: var(--dt-shadow-3);
  }
}
```

### 4.5 问题五：全局样式层存在重复定义和浏览器覆盖不足

**现状证据：**

- `global.scss` 中 `body`、`:root` 有重复覆盖。
- 滚动条样式以 `::-webkit-scrollbar` 为主，Firefox 无等价回退策略。

**风险：**

- 真实生效值难判断，调样式时需要反复追踪覆盖顺序。
- 跨浏览器一致性差，尤其是滚动条和输入态细节。

**推荐方案：**

- 把全局 token 只保留在 `design-tokens.scss`。
- `global.scss` 只做 reset、Element Plus override、实用类和跨浏览器补丁。
- 对滚动条提供渐进增强：WebKit 使用定制样式，Firefox 至少使用默认可见滚动条，不强做视觉一致。

## 7. 结论

前端页面样式与布局优化，不应被理解为“补几段 CSS”或“把页面做得更像设计稿”，而应视为一项系统工程：

- **规范层**：统一 token、骨架、状态与响应式约束。
- **实现层**：页面结构、滚动容器、组件边界、交互闭环。
- **性能层**：首屏加载、更新效率、资源利用率。
- **验证层**：视觉回归、交互回归、性能度量、兼容性核查。

后续如有前端优化任务，大模型应优先把本文件与 `35-前端全局样式与主题配置.md` 结合使用，避免再次产生"局部修样式、整体更混乱"的问题。

# 大模型 UI 像素级复刻实战指南

> **定位**：解决大模型在生成 Vue3/Element Plus 界面时脱离项目实际约束、凭空发散导致的设计偏差。
> **更新时间**：2026-03-21
> **核心原则**：一切 UI 实现必须且仅以 `13-需求明细.md` 和 `30-技术架构与约束.md` 为唯一数据契约，杜绝引入未定义的技术栈。

---

## 一、 当前实现与设计的偏差根源

大模型在当前项目中生成 UI 的实际偏差，主要因为忽略了已明确的高保真业务约束：
1. **聚合算法失效**：大模型习惯对所有消息渲染头像和默认间距，无视了《30-技术架构与约束》1.8.1 节中“连续 120s 内同发送者消息需隐藏头像并压缩间距为 4px”的强规。
2. **滚动逻辑粗糙**：大模型常采用 `overflow: auto` 的简陋处理，违背了 1.8.2 中要求的 `useVirtualList` 约束及“平滑滚动/上滑挂起提醒”的高级特性。
3. **输入区高度坍塌**：大模型喜欢写死高度，未执行 1.8.3 中要求的 `96px` 至 `300px` 弹性撑开规则。
4. **微交互生硬**：忽略了《13-需求明细》10.1 节对“Hover 时显露撤回/更多按钮、非 Hover 态洁净”的明确指示。

---

## 二、 严格受控的技术栈矩阵

为实现像素级对齐，大模型必须在以下限定矩阵内作业，**禁止推荐或引入未建档的工具（如未经核准的第三方花哨动效库等）**：
- **Tailwind 使用规范**：项目仅允许使用 `@apply` 工具类指令，禁止使用 `bg-primary-sidebar` 等 Tailwind 类名。具体规范见 [35-前端全局样式与主题配置.md](../35-前端全局样式与主题配置.md)。
- **基础视图**：Vue 3 `<script setup lang="js">` (严格遵守零 TypeScript 禁令)。
- **组件底层**：Element Plus。
- **自定义样式**：原生 Scoped CSS。对于 IM 核心面板（气泡、滚动窗），禁止使用 Element Plus 原生卡片凑数，必须手写符合 IM 布局的 Flex/Grid 结构。

---

## 三、 设计稿转化为代码的受控工作流

### 第一步：前置知识锁定 (Context Lock)
大模型在编写如“消息列表”、“聊天面板”等组件前，必须重新检索以下锚点文件，绝不凭“常识”写码：
- 查阅 `13-需求明细.md` (确认 UI 的加载策略、缩略图按 `200px` 逻辑)。
- 查阅 `30-技术架构与约束.md` (获取列表聚集、滚动控制蓝图)。

### 第二步：骨架结构生成 (Skeleton Render)
基于 `flex-direction: column` 构建全量结构：
- 对于图片气泡：必须默认配置 `<img>` 外包定宽定高容器，以阻断“加载抖动”。
- 对于输入框：必须开启 `@input` 高度监听，严格限制在 `min-height: 96px; max-height: 300px` 内运转，超出时触发局部滚动轴。

### 第三步：注入微交互规则 (Micro-Interaction)
确保组件生命周期内包含以下交互定义：
- 渲染 `PENDING` 业务状态时，剥离与服务端的时序耦合，实现“零延迟”上屏，辅以纯 CSS 轻微淡入。
- 建立 `.msg-bubble:hover .msg-actions` 的 CSS 悬浮穿透链，确保界面平时保持极简。

## 四、 具体实施策略与零幻觉控制点

### 4.1 样式变量强制溯源
大模型生成的代码中，若需使用间距或颜色，应优先在当前的 CSS 全局作用域中寻找以 `--dt-` 开头的钉钉系色值。如果找不到，则使用安全的中性色（如 `#FAFAFA`、`#E4E6EB` 等），**严禁使用红黄蓝等高饱和度通用颜色填充**。

### 4.2 严格防误伤重构
如果因为修复一个小按钮的颜色需要改动 CSS，大模型**只允许提供差分修改 (Diff)**，绝对禁止为了解决一个小问题而把整个 `<template>` 连带业务逻辑一起覆写，防止丢失绑定的 `traceId` 或状态机转换配置。

---

## 五、 UI 验证与测试标准 (Project Specific QA)

大模型交付代码后，系统/人工按照本项目的真实验收单执行：
- [ ] **TS 强禁查杀**：确保 `<script setup>` 没有遗漏 `lang="ts"`。
- [ ] **聚拢检查**：同一个人在 1 分钟内连发两条消息，第二条消息的头像是否确已隐藏，且气泡间隙是否等于 `4px`？
- [ ] **滚动检查**：快速发送 50 条消息，滚动条是否依然保持平滑，且是否按照 1.8.2 的要求并未出现性能卡顿？
- [ ] **回滚防护**：如果消息发送因后端报错 (模拟抛出异常)，气泡是否能够精准回滚并呈现“同步失败”红点提示（《13-需求明细》10.2）？

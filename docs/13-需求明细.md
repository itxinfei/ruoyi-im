# 13 需求明细

> 用途：把核心需求写成大模型可直接执行的明细。
> 原则：只写已知业务目标与明确规则；不知道的部分单独标记，不让模型自行补全。

## 1. 使用规则

1. 每个模块都按同一结构描述：
   - 功能目标
   - 用户入口
   - 前置条件
   - 主流程
   - 异常流程
   - 状态变化
   - 依赖与边界
   - 验收标准
   - 暂不实现
   - 已确认项/待确认项
2. 任务池应从本文件拆分任务，不反向定义需求。
3. 如果本文件未明确，模型必须去看 [04-待确认问题.md](04-待确认问题.md)，不能脑补。
4. **UI 验收强制要求**：默认体验参考对象是钉钉 Windows 客户端。**所有 UI 开发必须严格遵循 `docs/21-IM桌面端页面规范.md` 和 `docs/26-IM交互深度行为规范.md` 中定义的色值（#0089FF 主色）、8px 间距步进、气泡圆角及 Hover 交互规范**。
5. 本项目目标是私有化 Web 版钉钉，需求应从完整协同产品角度定义，而不只是单独 IM 页面。

## 2. 消息模块总说明

### 2.1 模块目标

消息模块是当前产品的主链路，目标：用户可完成以下动作：

1. 找到会话
2. 查看消息
3. 发送消息
4. 搜索消息
5. 处理协作动作（发送文件、@成员、引用回复）

实现原则：

1. 功能闭环优先于复杂架构。
2. 当前实现以单体 Java 后端和 Vue3 前端为前提。

### 2.2 模块边界

本阶段消息模块覆盖：

1. 会话列表
2. 聊天区
3. 输入区
4. 会话/消息搜索
5. 群详情
6. 群文件
7. 通话入口
8. 消息收藏

本阶段不要求：

1. 完整工作台协作闭环
2. 开放平台能力
3. 超出当前内网办公场景的社交能力

补充说明：

1. 尽管平台目标是完整钉钉式协同产品，但当前最优先细化的是 IM 主链路。
2. 左侧导航中其他模块也必须逐步补齐，不允许长期停留在占位状态。

### 2.3 平台级导航要求

### 2.3.1 功能目标

让产品具备钉钉式协同平台的一体化导航结构，而不是一个只有聊天区的局部应用。

### 2.3.2 左侧导航目标模块

1. 消息
2. 搜索
3. 通讯录
4. 工作台
5. 云盘/文档
6. 日历
7. 待办
8. 审批
9. 邮箱
10. AI 助手
11. 设置
12. 个人资料
13. 管理入口

### 2.3.3 验收标准

1. AC1：左侧导航展示的平台模块具有统一的视觉层级
2. AC2：每个已展示一级模块都存在真实页面
3. AC3：不允许长期保留无法进入或空白承接页

## 3. 会话列表

### 3.1 功能目标

用户可定位会话、识别未读、切换上下文。会话列表加载时间≤300ms。

### 3.2 用户入口

1. 进入消息首页后默认可见
2. 位于主页面左侧会话栏

### 3.3 前置条件

1. 用户已登录
2. 会话列表接口可返回当前用户会话

### 3.4 主流程

1. 页面加载后自动请求当前用户会话列表
2. 默认按最近活跃时间排序
3. 若存在置顶会话，置顶会话排在普通会话之前
4. 用户点击某个会话后，聊天区切换到该会话
5. 聊天区加载该会话最近消息

### 3.5 异常流程

1. 会话列表为空时，显示空状态，不显示报错占位文案
2. 会话列表加载失败时，显示明确失败提示
3. 会话被删除或失效时，当前会话状态应被重置

### 3.6 状态变化

1. 当前会话 `currentSession` 更新
2. 未读数在进入会话后按规则减少或清零
3. 置顶、免打扰等设置变化后，会话列表状态同步更新

### 3.7 依赖与边界

1. 依赖会话列表加载能力
2. 依赖当前会话状态管理
3. 会话列表 UI 保留：头像、名称、摘要、时间、未读数

### 3.8 验收标准

1. AC1：进入消息页后可看到当前用户的会话列表
2. AC2：点击任意会话后，聊天区切换到对应会话
3. AC3：未读数能够正确显示
4. AC4：空列表时显示空状态而不是异常文案

### 3.9 暂不实现

1. 多级复杂标签体系
2. 过度装饰的状态展示

### 3.10 已确认项

> ✅ 已确认（2026-03-20）

1. **未读清零规则**：用户可选（进入会话即清零 / 已读回执清零），需在设置中提供选项

## 4. 聊天区

### 4.1 功能目标

用户可查看消息上下文、浏览消息、定位消息、操作消息。

### 4.2 用户入口

1. 从会话列表点击某个会话进入

### 4.3 前置条件

1. 已有当前会话
2. 消息历史接口可用

### 4.4 主流程

1. 进入会话后加载最近消息
2. 默认滚动到底部
3. 若带有跳转消息 ID，则优先定位指定消息
4. 用户上滑时可继续加载历史消息
5. 鼠标悬停消息后，可执行回复、转发、复制、撤回等操作

### 4.5 异常流程

1. 历史消息加载失败时，提示失败且保留当前已显示消息
2. 跳转消息不存在时，提示未找到，不允许空白定位
3. 撤回失败时，原消息状态保持不变并提示失败

### 4.6 状态变化

1. 当前消息列表更新
2. 消息跳转高亮状态短时存在后消失
3. 加载历史消息时保持滚动位置稳定

### 4.7 依赖与边界

1. 依赖消息历史加载能力
2. 依赖消息操作接口
3. 聊天区不应承载与当前会话无关的信息

### 4.8 验收标准

1. AC1：进入会话后能看到最近消息
2. AC2：上滑可加载历史消息（必须采用虚拟长列表技术 `useVirtualList`，确保万条消息无白屏卡顿）。
3. AC3：引用跳转可以定位到原消息
4. AC4：撤回、转发、复制入口可触发真实行为（需严格遵循 `docs/ui-ux/01-消息核心模块交互与视觉规范.md` 中的 Hover 体验）。
5. AC5：聊天区必须进行组件化拆分设计（List、Bubble），气泡与头像基线必须对齐。

### 4.9 暂不实现

1. 复杂消息编排动画
2. 与当前业务无关的富媒体消息类型

### 4.10 消息编辑与已读详情业务规则

> ✅ 已确认（2026-03-20）

1. **消息编辑**：
   - **时间窗口**：发送后 5 分钟内允许编辑（保持不变）。
   - **编辑次数**：不限制编辑次数。
   - **状态显示**：编辑后消息下方显示"已编辑"灰色小字。
   - **历史追溯**：点击"已编辑"文字，弹出气泡显示"编辑历史"，展示历次修改内容及时间。
   - **多端同步**：通过 WebSocket 全局推送 `MSG_EDIT_NOTIFY` 消息，前端实时替换。
2. **已读详情**：
   - **触发入口**：消息气泡下方的"N人未读/全显已读"蓝色文字。
   - **分类展示**：弹窗内分"已读"和"未读"两个 Tab。
   - **成员展示**：展示成员头像、昵称，支持点击成员发起单聊。
3. **已读回执设置**：
   - **单聊可关闭**：用户可在设置中关闭单聊的已读回执功能。
   - **群聊强制开启**：群聊场景下已读回执不可关闭。

### 4.11 @提及与引用完整性 (Social Intelligence)
1. **强引用 Mentions**：
   - **数据格式**：发送消息时，除 `content` 外，需在 `atUserIds` 字段传入手动选择的用户 ID 列表。
   - **文本标记**：`content` 中的 `@xxx` 仅作为视觉展示，系统逻辑应以 `atUserIds` 数组为准。
   - **特殊提醒**：若包含 `@all`，需触发全局强提醒（即便用户开启了免打扰）。
2. **引用镜像 (Reference Snapshot)**：
   - **冗余存储**：发送引用消息时，前端应快照原消息的前 30 个字符存入 `replyQuote` 字段。
   - **失效处理**：若原消息被撤回，引用区应显示："原消息已撤回"，点击跳转动作失效，但快照内容依然保留以维持上下文连贯。

## 5. 输入区

### 5.1 功能目标

用户可发送文本、图片、文件消息。

### 5.2 用户入口

1. 进入任意会话后，在聊天区底部可见

### 5.3 前置条件

1. 当前会话存在
2. 发送消息接口可用

### 5.4 主流程

1. 用户输入文本并按 Enter 发送
2. 用户按 Ctrl+Enter 或 Cmd+Enter 换行
3. 用户点击图片或文件按钮后选择本地文件并发送
4. 用户粘贴图片或文件时，系统自动识别并进入上传流程
5. 用户直接将文件拖拽进入聊天面板，系统高亮提示并进入上传流程
6. 用户在群聊中输入 `@` 或点击 `@` 按钮时，拉起成员选择
7. 若存在回复上下文，则发送时带上回复关联信息

### 5.5 异常流程

1. 未选择会话时，不允许发送
2. 文件或图片超限时，提示限制原因
3. 上传失败时，消息保持失败状态并允许重试
4. URL 元数据解析失败时，降级为普通文本消息

### 5.6 状态变化与草稿逻辑

1. **输入草稿自动保存**：
   - 当用户在输入框输入内容但未发送，且切换到其他会话或关闭页面时，前端需自动将内容保存至本地/后端（`draft_content`）。
   - 再次切回该会话时，输入框自动回显内容。
   - 会话列表对应项显示红色文字 `[草稿] xxx`。
2. **清空逻辑**：发送成功后清空输入框与回复态，同时清除对应会话的草稿状态。
3. **上传状态**：上传中、失败、成功状态在气泡内应有明确反馈。

### 5.7 依赖与边界

1. 依赖消息发送能力
2. 依赖上传接口
3. 工具栏只放真实可用入口
4. 当前未确认录音入口闭环，语音消息不应伪装为已完成能力

### 5.8 验收标准

1. AC1：文本消息可发送（发送瞬间气泡必须呈现 PENDING 状态，展示 Loading 动画）。
2. AC2：图片与文件可发送
3. AC3：群聊可执行 @ 成员
4. AC4：回复消息可带上引用关系
5. AC5：粘贴上传可触发真实上传链路
6. AC6：弱网或发送超时（如5秒）状态变化为 FAILED，气泡展示红色预警并提供重试交互。

### 5.9 暂不实现

1. 富文本编辑器
2. 未闭环的录音按钮

### 5.10 语音消息接入规范

> ✅ 已确认（2026-03-20）

1. **交互流程**：点击工具栏录音图标 -> 弹出录音小窗 -> 松开/点击结束 -> 自动上传文件 -> 发送 VOICE 类型消息。
2. **最大时长**：语音消息最长 60 秒，超过自动停止录音并发送。
3. **展示效果**：气泡内显示波形图（或播放条）+ 时长，点击可播放/暂停。
4. **转文字**：支持右键点击"转文字"，调用后端语音转文本接口并在气泡下方展示结果。

## 6. 搜索 (IM会话内与全局)

### 6.1 功能目标
用户可在会话内定位消息，或全局快速检索联系人、群组。搜索响应≤500ms。

### 6.2 用户入口
1. 会话栏顶部搜索框 (全局)
2. 聊天头部搜索按钮 (会话内)

### 6.3 核心逻辑
- **会话内**：仅检索当前 `sessionId` 对应的消息记录。
- **全局**：同时检索 `im_conversation`、`sys_user`、`im_group`、`im_file`。
- **高亮**：搜索结果中关键词必须高亮显示。

### 6.4 验收标准
1. AC1：搜索结果支持点击并精准跳转定位。
2. AC2：无结果时显示明确占位，不返回空异常。

## 7. 协同辅助能力 (IM 衍生)

### 7.1 通话 (Call)
- **发起**：单聊/群聊头部发起，支持语音/视频。
- **状态**：calling -> connected -> ended。
- **规则**：断线 3s 内自动重连；已接通通话支持"最小化"悬浮窗展示。

### 7.2 消息收藏 (Favorites)
- **收藏**：右键消息气泡收藏。
- **列表**：左侧导航进入收藏面板。
- **跳转**：支持从收藏项跳回原始会话位置。

### 7.3 链接卡片 (Link Card)
- **解析**：系统自动识别 URL，调用 `/api/im/url/parse` 转为卡片。
- **降级**：解析失败或超时 (>3s) 则渲染为普通链接。

## 8. 开发约束与规范 (IM 专用)

### 8.1 状态机约束
- 发送流：PENDING -> SENDING -> SENT / FAILED。
- 撤回流：仅 SENT 且 < 2min。
- 编辑流：仅 SENT 且 < 5min。

### 8.2 数据一致性
- 必须确保 WebSocket 推送的消息与本地数据库状态同步。
- 已读回执在"单聊关闭"后，前端应隐藏该消息的已读/未读状态显示。

## 10. 微交互与鲁棒性准则 (IM 核心灵魂)

### 10.1 微交互准则 (UX Signature)
1. **瞬时反馈**：点击发送后，消息应立即进入聊天区（PENDING 状态），不等待后端返回，实现"零延迟"体感。
2. **气泡律动**：新消息进入时应有微小的位移/淡入过渡，禁止生硬刷新。
3. **自适应感知**：输入框高度应随内容自动平滑撑开，最大高度不超过聊天区 40%。
4. **悬浮态感知**：消息气泡在 Hover 时应显式展示"操作触发点"（如撤回/更多按钮），非 Hover 态保持洁净。

### 10.2 鲁棒性准则 (Armor Logic)
1. **消息去重**：前端必须根据 `clientMsgId` 执行严格去重，防止 WebSocket 与 HTTP 重复渲染。
2. **乐观更新策略**：编辑消息时先在本地替换内容，若后端报错则回滚并提示"同步失败"。
3. **断网生存**：网络离线时，发送动作应自动存入"本地重发队列"，待恢复后原子化重试。
4. **状态最终一致性**：WebSocket 推送的消息编辑/撤回指令，必须具备版本号校验，防止旧数据老旧覆盖新内容。

### 10.3 资源分级加载策略 (Resource Tiers)
1. **三级预览逻辑**：
   - **Thumbnails (缩略图)**：消息流中默认仅加载 200px 左右的极小缩略图，后端需支持 `?type=thumb` 参数。
   - **HD (高清图)**：用户在聊天区点击图片后，弹出层优先加载高清图（适配屏幕尺寸）。
   - **Original (原图)**：仅在用户点击"查看原图"或执行下载动作时，才请求 100% 原始文件。
2. **并发加载控制**：
   - 消息历史滚动时，图片加载必须具备 **"懒加载"** 机制，且同屏最大并发下载数不超过 6 个。
3. **过期清理 (TTL)**：
   - 前端本地缓存（IndexedDB）中的多媒体资源应具备 7 天 TTL，过期自动清理以释放磁盘空间。

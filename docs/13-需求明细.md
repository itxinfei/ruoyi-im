# 13 需求明细

> 用途：把核心需求写成大模型可直接执行的明细。
> 原则：只写已知业务目标与明确规则；不知道的部分单独标记，不让模型自行补全。

## 1. 使用规则

1. 每个模块都按同一结构描述：
   - 功能目标
   - 用户入口
   - 前置条件
   - 主流程
   - 异常流程
   - 状态变化
   - 依赖与边界
   - 验收标准
   - 暂不实现
   - 已确认项/待确认项
2. 任务池应从本文件拆分任务，不反向定义需求。
3. 如果本文件未明确，模型必须去看 [04-待确认问题.md](04-待确认问题.md)，不能脑补。
4. **UI 验收强制要求**：默认体验参考对象是钉钉 Windows 客户端。**所有 UI 开发必须严格遵循 `docs/21-IM桌面端页面规范.md` 和 `docs/26-IM交互深度行为规范.md` 中定义的色值（#277EFB 主色）、8px 间距步进、气泡圆角及 Hover 交互规范**。
5. 本项目目标是私有化 Web 版钉钉，需求应从完整协同产品角度定义，而不只是单独 IM 页面。

## 2. 消息模块总说明

### 2.1 模块目标

消息模块是当前产品的主链路，目标：用户可完成以下动作：

1. 找到会话
2. 查看消息
3. 发送消息
4. 搜索消息
5. 处理协作动作（发送文件、@成员、引用回复）

实现原则：

1. 功能闭环优先于复杂架构。
2. 当前实现以单体 Java 后端和 Vue3 前端为前提。

### 2.2 模块边界

本阶段消息模块覆盖：

1. 会话列表
2. 聊天区
3. 输入区
4. 会话/消息搜索
5. 群详情
6. 群文件
7. 通话入口
8. 消息收藏

本阶段不要求：

1. 完整工作台协作闭环
2. 开放平台能力
3. 超出当前内网办公场景的社交能力

补充说明：

1. 尽管平台目标是完整钉钉式协同产品，但当前最优先细化的是 IM 主链路。
2. 左侧导航中其他模块也必须逐步补齐，不允许长期停留在占位状态。

### 2.3 平台级导航要求

### 2.3.1 功能目标

让产品具备钉钉式协同平台的一体化导航结构，而不是一个只有聊天区的局部应用。

### 2.3.2 左侧导航目标模块

1. 消息
2. 搜索
3. 通讯录
4. 工作台
5. 云盘/文档
6. 日历
7. 待办
8. 审批
9. 邮箱
10. AI 助手
11. 设置
12. 个人资料
13. 管理入口

### 2.3.3 验收标准

1. AC1：左侧导航展示的平台模块具有统一的视觉层级
2. AC2：每个已展示一级模块都存在真实页面
3. AC3：不允许长期保留无法进入或空白承接页

## 3. 会话与联系人列表 (L2 Session Bar)

### 3.1 功能目标
让用户在左侧列表快速定位聊天目标。列表应包含：
- **置顶会话**：固定在顶部，背景色略深。
- **普通会话**：按最后消息时间排序。
- **未读提醒**：红色角标显示未读数。
- **联系人数据**：支持从通讯录直接拉取联系人进入会话。

### 3.2 UI 规格 (DingTalk Spec)
- **色值对齐**：选中项背景使用 `#E6F2FF`，激活蓝边使用 `#277EFB`（对齐登录页主色）。
- **组件结构**：Avatar + Title (Name) + Subtitle (Last Msg) + Time + Badge.

## 4. 聊天区与消息内容 (L3 Chat Panel)

### 4.1 功能目标
展示核心聊天内容。结构如下：
1. **聊天头部 (Chat Header)**：显示当前聊天对象/群组名称，右侧为详情开关。
2. **消息列表 (Message List)**：垂直滚动的消息气泡流。
3. **输入区 (Input Area)**：位于底部，支持多种消息类型发送。

### 4.2 数据展现 (Data Flow)
- **数据来源**：调用 `/api/im/message/list` 拉取分页消息。
- **实时性**：通过 WebSocket `MSG_PUSH` 帧实时渲染新消息。
- **空状态**：未选中会话时，L3 区域显示居中的 Logo 与“欢迎使用”占位插画（对齐登录页视觉）。

### 4.3 视觉要求
- **气泡对齐**：发送方（右侧蓝色气泡 `#277EFB`），接收方（左侧白色气泡）。
- **背景色**：聊天区域背景统一使用 `#F2F4F5`。

## 5. 输入区

### 5.1 功能目标

用户可发送文本、图片、文件消息。

### 5.2 用户入口

1. 进入任意会话后，在聊天区底部可见

### 5.3 前置条件

1. 当前会话存在
2. 发送消息接口可用

### 5.4 主流程

1. 用户输入文本并按 Enter 发送
2. 用户按 Ctrl+Enter 或 Cmd+Enter 换行
3. 用户点击图片或文件按钮后选择本地文件并发送
4. 用户粘贴图片或文件时，系统自动识别并进入上传流程
5. 用户直接将文件拖拽进入聊天面板，系统高亮提示并进入上传流程
6. 用户在群聊中输入 `@` 或点击 `@` 按钮时，拉起成员选择
7. 若存在回复上下文，则发送时带上回复关联信息

### 5.5 异常流程

1. 未选择会话时，不允许发送
2. 文件或图片超限时，提示限制原因
3. 上传失败时，消息保持失败状态并允许重试
4. URL 元数据解析失败时，降级为普通文本消息

### 5.6 状态变化与草稿逻辑

1. **输入草稿自动保存**：
   - 当用户在输入框输入内容但未发送，且切换到其他会话或关闭页面时，前端需自动将内容保存至本地/后端（`draft_content`）。
   - 再次切回该会话时，输入框自动回显内容。
   - 会话列表对应项显示红色文字 `[草稿] xxx`。
2. **清空逻辑**：发送成功后清空输入框与回复态，同时清除对应会话的草稿状态。
3. **上传状态**：上传中、失败、成功状态在气泡内应有明确反馈。

### 5.7 依赖与边界

1. 依赖消息发送能力
2. 依赖上传接口
3. 工具栏只放真实可用入口
4. 当前未确认录音入口闭环，语音消息不应伪装为已完成能力

### 5.8 验收标准

1. AC1：文本消息可发送（发送瞬间气泡必须呈现 PENDING 状态，展示 Loading 动画）。
2. AC2：图片与文件可发送
3. AC3：群聊可执行 @ 成员
4. AC4：回复消息可带上引用关系
5. AC5：粘贴上传可触发真实上传链路
6. AC6：弱网或发送超时（如5秒）状态变化为 FAILED，气泡展示红色预警并提供重试交互。

### 5.9 暂不实现

1. 富文本编辑器
2. 未闭环的录音按钮

### 5.10 语音消息接入规范

> ✅ 已确认（2026-03-20）

1. **交互流程**：点击工具栏录音图标 -> 弹出录音小窗 -> 松开/点击结束 -> 自动上传文件 -> 发送 VOICE 类型消息。
2. **最大时长**：语音消息最长 60 秒，超过自动停止录音并发送。
3. **展示效果**：气泡内显示波形图（或播放条）+ 时长，点击可播放/暂停。
4. **转文字**：支持右键点击"转文字"，调用后端语音转文本接口并在气泡下方展示结果。

## 6. 搜索 (IM会话内与全局)

### 6.1 功能目标
用户可在会话内定位消息，或全局快速检索联系人、群组。搜索响应≤500ms。

### 6.2 用户入口
1. 会话栏顶部搜索框 (全局)
2. 聊天头部搜索按钮 (会话内)

### 6.3 核心逻辑
- **会话内**：仅检索当前 `sessionId` 对应的消息记录。
- **全局**：同时检索 `im_conversation`、`sys_user`、`im_group`、`im_file`。
- **高亮**：搜索结果中关键词必须高亮显示。

### 6.4 验收标准
1. AC1：搜索结果支持点击并精准跳转定位。
2. AC2：无结果时显示明确占位，不返回空异常。

## 7. 协同辅助能力 (IM 衍生)

### 7.1 通话 (Call)
- **发起**：单聊/群聊头部发起，支持语音/视频。
- **状态**：calling -> connected -> ended。
- **规则**：断线 3s 内自动重连；已接通通话支持"最小化"悬浮窗展示。

### 7.2 消息收藏 (Favorites)
- **收藏**：右键消息气泡收藏。
- **列表**：左侧导航进入收藏面板。
- **跳转**：支持从收藏项跳回原始会话位置。

### 7.3 链接卡片 (Link Card)
- **解析**：系统自动识别 URL，调用 `/api/im/url/parse` 转为卡片。
- **降级**：解析失败或超时 (>3s) 则渲染为普通链接。

## 8. 开发约束与规范 (IM 专用)

### 8.1 状态机约束
- 发送流：PENDING -> SENDING -> SENT -> DELIVERED / FAILED。
  - 状态说明：PENDING(待发送) -> SENDING(发送中) -> SENT(已发送) -> DELIVERED(已送达) / FAILED(发送失败)
- 撤回流：仅 SENT 且 < 2min。
- 编辑流：仅 SENT 且 < 5min。

### 8.2 数据一致性
- 必须确保 WebSocket 推送的消息与本地数据库状态同步。
- 已读回执在"单聊关闭"后，前端应隐藏该消息的已读/未读状态显示。

## 10. 微交互与鲁棒性准则 (IM 核心灵魂)

### 10.1 微交互准则 (UX Signature)
1. **瞬时反馈**：点击发送后，消息应立即进入聊天区（PENDING 状态），不等待后端返回，实现"零延迟"体感。
2. **气泡律动**：新消息进入时应有微小的位移/淡入过渡，禁止生硬刷新。
3. **自适应感知**：输入框高度应随内容自动平滑撑开，最大高度不超过聊天区 40%。
4. **悬浮态感知**：消息气泡在 Hover 时应显式展示"操作触发点"（如撤回/更多按钮），非 Hover 态保持洁净。

### 10.2 鲁棒性准则 (Armor Logic)
1. **消息去重**：前端必须根据 `clientMsgId` 执行严格去重，防止 WebSocket 与 HTTP 重复渲染。
2. **乐观更新策略**：编辑消息时先在本地替换内容，若后端报错则回滚并提示"同步失败"。
3. **断网生存**：网络离线时，发送动作应自动存入"本地重发队列"，待恢复后原子化重试。
4. **状态最终一致性**：WebSocket 推送的消息编辑/撤回指令，必须具备版本号校验，防止旧数据老旧覆盖新内容。

### 10.3 资源分级加载策略 (Resource Tiers)
1. **三级预览逻辑**：
   - **Thumbnails (缩略图)**：消息流中默认仅加载 200px 左右的极小缩略图，后端需支持 `?type=thumb` 参数。
   - **HD (高清图)**：用户在聊天区点击图片后，弹出层优先加载高清图（适配屏幕尺寸）。
   - **Original (原图)**：仅在用户点击"查看原图"或执行下载动作时，才请求 100% 原始文件。
2. **并发加载控制**：
   - 消息历史滚动时，图片加载必须具备 **"懒加载"** 机制，且同屏最大并发下载数不超过 6 个。
3. **过期清理 (TTL)**：
   - 前端本地缓存（IndexedDB）中的多媒体资源应具备 7 天 TTL，过期自动清理以释放磁盘空间。

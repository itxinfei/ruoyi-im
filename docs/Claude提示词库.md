```text
7. backend/01-消息模块后端代码重构与规范.md

9. backend/03-日志掩码与可观测性规约.md

11. ui-ux/01-消息核心模块交互与视觉规范.md
【任务派发】
12. 指定任务卡或 llm-tasks.yaml 中的单个 task_id

要求：
1. 只规划一个 task_id
2. 输出实现步骤，最多 7 步
3. 输出改动文件列表
4. 输出验收条目对照
5. 输出风险与回滚方案
6. 如果信息缺失，**必须立即触发 HALT 停机阻断**，严禁依靠常识猜测或捏造 Mock 数据框架。
7. 默认产品目标是"私有化部署的 Web 版钉钉"，默认体验参考钉钉 Windows 客户端
8. 如果任务与 IM 主链路冲突，IM 优先
9. 在设计任何新技术方案前，强制要求先执行 Glob/Grep 搜索工作区，绝不允许凭空新建已存在的工具类或同名 Bean。
10. 所有的前端样式严格限制在 CSS Tokens 库内取值，严禁 AI 自由编造十六进制颜色。
```

## 2. 开发提示词

```text
执行一个单任务开发。

输入（必读基准）：
1. 02-大模型研发规范.md (绝对红线、命名清洗、类型降级、无TS原则)
2. 30-技术架构与约束.md
3. 13-需求明细.md
4. 31-实体与状态模型.md
5. 17-接口与入口清单.md
6. backend/01-消息模块后端代码重构与规范.md
7. backend/02-全局错误码与异常字典规范.md
8. backend/03-日志掩码与可观测性规约.md
9. frontend/01-消息模块Vue组件化与性能优化.md
10. ui-ux/01-消息核心模块交互与视觉规范.md
11. 具体任务卡

硬约束重点执行：
1. 只改 scope_in
2. 绝对不可使用 TypeScript！严格依据《02-大模型研发规范.md》写纯 JS `<script setup lang="js">`。
3. 极速阻断原则：凡是未在上下文中找到的 API 方法或变量定义，**立刻停机**并向用户抛出疑问，严禁凭借名称相似度自行捏造 `handleXxx` 等占位符。必须彻底清除所有无意义命名。
4. 后端必须完全遵照 JDK 1.8，不用 `var`、不吞异常，返回接口严格遵循 `DTO`/`VO` 分层结构。
5. 必须运行 npm run build 并报告结果
6. 目标形态是"私有化 Web 版钉钉"，UI、布局、交互优先对齐钉钉 Windows 客户端，禁止自行造多余特效或动画。
7. 建立组件或写配置前，必须使用搜索工具 (Grep/Glob) 全局核实是否重复，执行防重合并原则。
8. UI 开发强制打上“瞎子滤镜”：完全依靠调用 CSS Tokens (如 var(--el-color-primary)) 涂装，**绝对剥夺**强写十六进制颜色代码 (如 #333333) 的权力。
9. 强制内存防泄漏规约：必须确保前端组件在 onUnmounted 中有完整的定时器与全局监听 (mitt.off, ws.off) 摧毁闭环。

输出：
1. 改动文件
2. 关键实现
3. 验收对照
4. 验证结果
5. 风险与遗留项
```

## 3. 审查提示词

```text
请审查本次任务改动，重点找问题而不是总结。

输入：
1. 任务卡
2. 改动文件
3. 02-大模型研发规范.md (审核核心：是否出现了 TypeScript、垃圾命名、Java9+结构、裸异常、打印明文)
4. 13-需求明细.md
5. backend/ 以及 frontend/ 等目录下的纵深防爆点指引文档。

要求：
1. 按 P0/P1/P2 输出问题
2. 每个问题必须包含文件路径、触发条件、风险后果、修复建议
3. 若无阻断问题，明确写"未发现阻断问题"
4. 最后列出测试盲区
5. 额外检查是否偏离钉钉 Windows 客户端的布局、层级、交互节奏
```

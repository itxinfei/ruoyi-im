# 26 IM 交互深度行为规范

> **定位**：定义 IM 核心页面的动态交互逻辑，严禁"静态平铺"式设计，确保符合桌面端软件使用习惯。
> **核心原则**：触发式展示、层级化交互、状态无损切换。

---

## 1. 详情面板展示规范 (The Drawer Principle)

### 1.1 严禁“常驻右侧”
- **错误行为**：聊天区右侧始终显示一个占位符或静态的详情栏。
- **正确行为**：详情栏必须是 **`el-drawer` (侧边抽屉)** 或 **`el-dialog` (弹窗)**。
- **触发规则**：
    - **单聊详情**：点击聊天头部（ChatHeader）的对方头像或昵称触发，从右侧滑出。
    - **群聊详情**：点击聊天头部右上角的 `...` 或群名称触发，从右侧滑出。
    - **个人资料**：点击左侧导航栏底部头像触发，弹出居中弹窗。

### 1.2 详情面板交互
- 详情面板弹出时，**不应压缩**中间聊天区的宽度（除非屏幕极宽 > 1600px），默认应采用 `overlay` (遮罩层) 模式或平滑推开。

---

## 2. 消息流交互规范 (Message Flow)

> 核心要求：必须将交互感知与底层 `ChatMessageBubble.vue` 组件化状态深度绑定。

### 2.1 悬停感知 (Hover Awareness)
- **错误行为**：消息旁边的“复制、撤回、转发”按钮始终显示。
- **正确行为**：
    - 默认隐藏操作栏。
    - 鼠标 `hover` 在单条消息气泡上时，在气泡上方或侧方**立即**浮现操作工具条，去除过渡和淡入淡出动画。
    - 离开后立即消失。

### 2.2 气泡与头像基线对齐 (Alignment)
- **红线**：头像顶部必须与消息气泡的第一行文字顶部对齐，严禁垂直居中。此规定对齐真实企服标准体验。

### 2.3 消息发送状态闭环 (Sending Status)
- **状态流转机制**：必须配合 Pinia 缓存实现“乐观更新”（Optimistic UI）。
  - **发送瞬间 (PENDING)**：气泡立刻上屏展现，并在外侧展示简单的“发送中...”文字或静态小图标，严禁开发复杂的 CSS 旋转动画。
  - **接口确认 (SENT)**：收到后台 HTTP 响应或 WebSocket 回执（ACK）后，发送状态文本立即消失。
  - **内网弱网超时 (FAILED)**：若设定时间（如 5000ms）未收到回执，转变为鲜明的红色警告图标 `!` 或“发送失败”文字。此时客户端的 `offlineQueue`（离线重发队列）必须提供**点击重推**交互。

### 2.4 已读/未读心智机制 (Read Receipts)
- 状态标记仅出现于发起方视图。
- 采用低对比度文字（浅蓝色“XX人未读”，或单纯的淡灰“已读”）以降低视觉噪音。
- 在接收到后端 WS 推送的批量已读信令后，前端直接替换对应的文本，不使用复杂的 CSS 过渡。

---

## 3. 会话切换交互 (Context Switching)

### 3.1 状态保持
- **要求**：切换会话时，消息列表滚动位置必须保持，输入框内容（草稿）必须保留。
- **加载反馈**：切换瞬间使用基础的全局 Loading 或局部文字提示即可，避免要求大模型开发复杂的 `Skeleton` (骨架屏)。

---

## 4. 平台级模块切换 (Primary Nav)

### 4.1 独立路由空间
- 每个一级模块（消息、通讯录、工作台）拥有独立的布局空间。
- 切换模块时，左侧一级导航栏（窄栏）固定不动，中间和右侧区域进行 `Fade` 切换。

---

## 5. UI 验收负面清单 (Negative Constraints)

1. **禁止**：使用超过 `12px` 的大圆角（IM 应显得专业、稳重，采用 `4px-8px` 圆角）。
2. **禁止**：在桌面端使用偏移动端的“底部 Tab 栏”。
3. **禁止**：按钮使用过大的 `Padding` 或占据整行的宽度（除非是登录页）。
4. **禁止**：在未连接 WebSocket 时显示任何“假”的在线状态。

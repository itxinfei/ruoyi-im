# 02 大模型研发规范

> **定位**：定义大模型/AI 开发时的绝密法则、动作红线与质量审批拦截标准。
> **更新时间**：2026-03-21
> **重要性**：全目录最高（覆盖所有业务约束）。必须在此宪法体系之上开展开发。

---

## 绝对红线法则 (Zero-Tolerance Rules)

### 1. JDK 1.8 语法铁律
**强制 1.8，禁用任何 Java 9+ 语法特性。** 一旦发现直接判定产出为废品。
- **不用 `var`**：禁止类型推断语句。
- **不用 `record` / `sealed`**：仅准许使用 `@Data` 注解普通的 `class` 实体。
- **不用 `switch ->` 语法**：只允许使用传统的 `case:` 与 `break;` 穿透块。
- **不用 `"""` 文本块**：必须使用标准的字符串拼接。
- **禁用 `isEmpty()` / `toList()`**：使用传统的非空判断以及 `Collectors.toList()`。
- **禁用 Pattern matching instanceof**：不准用 `if (obj instanceof User user)`，必须老实强转。

### 2. Vue 3 规范铁律
**强制无 TypeScript 的纯 JS Component 化设计。**
- **必须使用 `<script setup lang="js">` 语法**，严禁出现 `<script setup lang="ts">` 或引入 `interface/type/enum` 特性。
- CSS 必须全部标明 `<style scoped>` 以防止全局污染。
- 只有被明确定义为“页面级别”的文件方可放置在 `views/` 目录下，所有与 IM 会话、消息气泡有关的细粒度抽离必须写在 `components/im/` 里。

### 3. 三层架构隔离不可逾越
```text
Controller (接收参数、调 Service、VO出)
   ↓ 单向依赖
Service (业务逻辑处理、事务 @Transactional)
   ↓ 单向依赖
Mapper (Mybatis-Plus 单表/多表装配操作)
```
- **禁止 Entity 作为 API 入参或返回值**，出入参必须走 `DTO`/`VO` 对象，依靠 `Controller` 层来完成 `BeanUtil.copyProperties`。
- **没有业务逻辑直接调用 Mapper**，即使是非常简单的根据主键查询，也必须老实地走 `Service` 层方法。

### 4. 数据隔离与查询红线
- **必须使用 MyBatis-Plus Lambda 风格查询代码**：`wrappers.lambdaQuery().eq(ImUser::getId, userId)`，严厉封杀使用任何形似 `eq("id", userId)` 的硬编码魔术字符串！
- 大单表（如消息历史表）只要查询，**必须分页** 或 限定 `LIMIT`（单页上限≤1000）。

### 5. 垃圾代码零容忍与命名肃清法则
- 命名禁区：**禁止用无意义词汇作为变量名或方法首签**。
    - ❌禁止：变量名为 `data` -> 应当为 `messageVO` 或 `authDTO`。
    - ❌禁止：变量名为 `info` -> 应当为 `userDetailInfo`。
    - ❌禁止：变量名为 `temp` -> 应当为 `buffer`。
    - ❌禁止：**方法前缀叫 `handle`**（非常多大模型的恶习） -> 应当改为动词如 `processMessage`、`executeSearch`、`validateUser` 等。
- 不提供无意义的方法：绝不创建"未来可能用到"的代码；如果可以依靠 `Hutool` 和 `Apache Commons` 完成的判断不要自行抽离 Utils。

## 控制边界与拦截门 (Quality Gates)

### 1. 代码准入红线
- **不能吞异常**：无论何种不可预见的 `Exception`，严禁一个空 Catch 略过。所有报错由 `@RestControllerAdvice` 收敛拦截（见 `02-全局错误码与异常字典规范.md`）。
- **可观测性合规**：无论是前端 Console 还是后端 Log，均**绝不允许明文打印消息的正文 `content`、`token` 和密钥**，相关调试信息脱敏处理。必须满足对核心行为（接入、报错、断开）的拦截记录（见 `03-日志掩码与可观测性规约.md`）。

### 2. 计划与存量复用基准
- 大于等于 **2** 个文件的逻辑重构、新功能添加，不论看起来多么普通，在实际开写代码前必须用 `implementation_plan.md` 陈述出修改的层级。
- 绝不重复造轮子！对于认证体系和全局工具，动手生成前必须先执行文件搜索 (`Grep` / `Glob`) 看此项目中是否已存在。

## 高可用与防御式编程准则 (High-Availability & Defensive Thinking)

### 1. 幂等性与并发安全
- **后端幂等**：所有涉及状态变更（撤回、编辑、审批）的接口，必须具备幂等处理能力。通过 `clientMsgId` 或 `version` 字段校验重复请求。
- **前端时序锁**：所有提交动作（点击发送、点击审批）必须具备 **"防重点击"** 机制（Button Loading 态），严禁连续点击触发多次请求。

### 2. 状态最终一致性
- **降级容灾**：WebSocket 推送仅作为"实时增强"。AI 在实现逻辑时必须确保：即便 WebSocket 彻底失效，用户通过手动刷新页面（触发 REST 列表查询）依然能看到 100% 准确的业务状态。
- **数据对齐**：前端 Store（如 Pinia）中的数据更新，必须以服务端返回的 `seq` 或 `timestamp` 为准，严禁仅凭前端推测更新本地状态。

### 3. 异常上下文保留
- **有价值的日志**：抛出 `BusinessException` 时，必须带上受影响的 `id` 或 `code`。严禁抛出类似 "系统错误" 这种不带任何调试线索的垃圾异常。

---

### 3. 极速阻断与零幻觉法则 (Zero Hallucination)
- **绝对禁止脑补API/方法**：当你需要调用后端服务、操作缓存、或复用组件时，如果当前上下文中尚未看到被调用的源码文件，**必须先使用搜索工具去查找文件并验证其存在**！绝不允许仅仅因为“听起来项目中应该有”就凭空捏造一个不存在的 `UserUtils.getCurrentId()` 甚至自编一个 `pom.xml` 依赖项。
- **未知即停机 (Unknown = Halt)**：遇到任何逻辑断层、缺失的关键字段、或者需求交叉矛盾。**不准输出猜测的代码，不准写入“占位符”逻辑**，立即停止执行并呼叫用户说明：“这里存在信息缺失，我无法脑补，请提供明确指令”。
- **严禁编写“未来可能用到”的功能**：如果用户要求写 A 功能，只写 A 功能。不要顺手写一个未被要求的 B 功能，也不要留下大段大段的 `// TODO:` 注释块。
- **全域唯一真相来源 (SSOT)**：业务规则必须且只能从 `12-需求点清单.md` 的 `C-xxx` 池中提取。不准凭记忆去猜“消息应该多少分钟撤回”。

## 架构冲突预防与 SSOT 扫描红线 (Conflict Prevention)

为了防止 Bean 名称冲突（如 `ConflictingBeanDefinitionException`）及功能重复实现，AI 必须严格执行：

### 1. 强制性前置扫描 (Pre-creation Scan)
- **创建类之前**：必须执行 `Glob` 或 `Grep` 搜索整个 `src/` 目录，确认是否已存在功能相似的类或同名 Bean。
- **配置合并原则**：若项目中已存在相关模块的配置类（如 `WebSocketConfig`），**严禁新建**，必须将新逻辑通过方法注入或接口实现的方式合并至原有类中。

### 2. 命名冲突拦截
- **Bean 唯一性**：严禁在不同包下创建同名的 `@Configuration`、`@Component` 或 `@Service`。
- **自动装配警惕**：若发现自动装配失败，必须第一时间检查是否由于自己新建了重复的 Bean 导致容器解析混乱。

### 3. 拦截器与工具类复用
- 优先查找 `util`, `config`, `common` 包下的存量工具。严禁为了方便而随手新建 `xxxUtils`。

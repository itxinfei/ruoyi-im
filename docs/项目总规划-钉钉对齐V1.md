# RuoYi-IM 项目总规划（钉钉对齐 V1）

> 目标：直接作为后续开发唯一执行规划，不再依赖其他大模型二次规划。  
> 对标基线：钉钉 8.2（主对标）+ 飞书/野火IM（补充参考）。  
> 技术栈硬约束：JDK8、Vue3、MySQL5.7、开发 Win11、生产 CentOS7。

---

## 1. 规划目标（V1）

1. 先做“聊天可替代”：消息主链路达到可日常办公使用。
2. 再做“组织可协同”：通讯录、群组、工作台入口闭环。
3. 最后做“企业可运营”：管理、审计、稳定性与可维护性。

---

## 2. V1 冻结范围（必须做）

### 2.1 一级导航（Web）

1. 消息
2. 通讯录
3. 工作台

### 2.2 二级菜单（必须完整可用）

1. 消息域：全部、未读、置顶、免打扰、群聊、文件链接
2. 通讯录域：组织架构、我的好友、群组管理、最近联系人
3. 工作台域：待办、审批、日程、公告、文档入口

### 2.3 三级弹窗（必须统一）

1. 消息域：发起会话、会话详情、文件预览、语音/视频通话、消息搜索
2. 通讯录域：联系人详情、添加好友、创建群组、成员管理
3. 工作台域：新建待办、发起审批、发布公告、日程编辑

统一规则：标题区+内容区+操作区；支持 ESC、右上角关闭、取消按钮关闭；提交按钮有 loading 和失败提示。

---

## 3. 功能规划（你将开发的功能）

## 3.1 P0（必须先完成）

1. 会话列表与筛选（对应二级菜单）
2. 文本/图片/文件消息发送与重试
3. 历史消息分页加载（lastId/limit 统一）
4. 消息状态（发送中/成功/失败）与已读回执
5. 回复/转发/撤回/删除
6. 会话置顶、免打扰、标记已读
7. 发起单聊/建群与会话详情弹窗

## 3.2 P1（第二阶段）

1. 全局消息搜索与会话内搜索
2. @提及、消息收藏、待办关联
3. 群公告、群文件、群成员管理
4. 语音/视频通话（WebRTC）可用闭环

## 3.3 P2（优化阶段）

1. 快捷键体系与效率增强
2. 性能优化（长列表虚拟化、懒加载）
3. 多端一致性体验与异常降级优化

---

## 4. 消息模块 UI 设计定版（核心）

### 4.1 页面骨架（钉钉式三段）

1. 左：会话区（含二级菜单切换 + 搜索）
2. 中：聊天区（头部、消息流、输入区）
3. 右：详情区（会话信息/群信息，抽屉或侧栏）

### 4.2 交互主路径

1. 进入消息 -> 默认选中首会话 -> 自动加载消息
2. 输入发送 -> 乐观渲染 -> 成功回填真实消息 -> 失败可重试
3. 上滑加载历史 -> 保持滚动锚点不跳动
4. 右键/悬浮菜单 -> 回复/转发/收藏/删除

### 4.3 可用性红线

1. 不允许只有样式无业务动作。
2. 不允许出现“点击无响应”的二级菜单或弹窗入口。
3. 不允许在主链路出现“开发中”占位。

---

## 5. 前后端设计与接口冻结

## 5.1 API 约定（关键）

1. 会话列表：`GET /api/im/conversation/list`
2. 消息列表：`GET /api/im/message/list/{conversationId}`  
   参数统一：`lastId`、`limit`（兼容 `lastMessageId`、`pageSize`）
3. 发送消息：`POST /api/im/message/send`
4. 已读上报：`PUT /api/im/message/read`

## 5.2 字段统一（关键）

1. 前端消息类型字段统一用 `type`，后端内部可映射 `messageType`
2. 编辑消息请求统一落到 `newContent`
3. 转发目标会话统一落到 `toConversationId`
4. 所有时间字段返回 `timestamp` 兼容前端渲染

---

## 6. 数据库与 Mapper 对齐规划（以 `sql/im.sql` 为准）

1. 单一事实源：`sql/im.sql` 为最终表结构标准。
2. 每次表结构变更后，必须执行三步：
   - 更新实体（domain）
   - 更新 Mapper XML/Mapper 接口
   - 更新 DTO/VO 与 Service 映射
3. 建立“差异检查清单”：
   - 字段名一致
   - 类型一致
   - 非空/默认值一致
   - 索引与查询条件一致
4. 禁止“表已改、Mapper未改、接口继续跑”的漂移状态。

---

## 7. 开发执行模式（单模型直驱）

1. 规划、开发、审查由当前模型直接完成，不再切换其他模型。
2. 每次任务必须提供：
   - 目标
   - 改动文件
   - 验证命令
   - 验收结论
3. 若需求变更，先改文档再改代码，保持“文档先行”。

---

## 8. 代码规范（强制）

### 8.1 Java（JDK8）

1. 禁止 Java 9+ 语法（`var`、`record`、switch 新语法等）。
2. 禁止魔法值，统一常量/枚举。
3. 禁止复杂表达式链，优先 `if/for` 显式逻辑。
4. 方法可读优先，超长方法必须拆分。

### 8.2 Vue3

1. 仅使用 Vue3（Composition API）。
2. API 请求必须集中封装，禁止组件内散乱拼接口。
3. UI 状态与业务状态一致，失败必须可见。
4. 禁止重复造轮子与重复创建同类文件。

---

## 9. 里程碑计划（4 周）

1. M1（第1周）：消息 P0 全链路 + 二级菜单完整可达
2. M2（第2周）：三级弹窗标准化 + 通讯录主路径
3. M3（第3周）：工作台主路径 + 管理端必要能力
4. M4（第4周）：联调修复、性能优化、验收回归

每周交付都必须满足：可构建、可演示、可验收。

---

## 10. 验收标准（发布前）

1. 主路径可用：登录 -> 选会话 -> 发消息 -> 收消息 -> 搜索 -> 管理会话
2. 二级菜单全部可用，无死入口
3. 三级弹窗全部可提交，无样式空壳
4. 前端构建通过，后端打包通过
5. 文档、代码、数据库结构三者一致

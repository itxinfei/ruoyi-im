## 一、项目目标与范围

### 1.1 产品目标

1. 在企业内网场景下，提供稳定、低延迟、高可用的即时通讯能力。
2. 在主流程体验上达到“钉钉可替代级”：
   - 找人快
   - 发消息快
   - 看消息清晰
   - 协作入口不绕路
3. 工作台模块达到“可日常使用级”，不是只展示。

### 1.2 目标用户

1. 10-500人规模企业员工（内网办公）
2. 组织管理员（权限、组织、审计）
3. 部门负责人（群组、审批、待办协作）

### 1.3 非目标（当前版本不追求）

1. 公网社交属性
2. 开放生态应用市场完整复刻
3. 超大规模万人并发架构

## 二、技术与部署基线

### 2.1 技术栈（强约束）

| 层 | 技术栈 | 约束版本 |
|------|------|------|
| 前端 | Vue 3 + Vite + Element Plus + Vue Router + Vuex/Pinia + WebSocket | Vue `3.x`，Node `18 LTS` |
| 后端 | Spring Boot + Spring Security + JWT + MyBatis Plus | **JDK `1.8` 必须** |
| 数据层 | MySQL + Redis + 本地文件系统 | MySQL `5.7`，Redis `6+` |
| 实时音视频 | WebRTC | 只允许走浏览器标准能力，不引入不必要SDK |

补充约束：

1. 不允许提交任何 Java 9+ 语法。
2. 不允许降级到 Vue2 或 Options API 大规模回退。
3. 前后端接口必须通过 DTO/VO 映射，不允许 Entity 直出。
4. 状态类数据统一优先写入 `im_user_session`（详见《数据库统一与迁移规范.md》）。
5. Java 代码禁止硬编码业务常量（魔法值），必须抽常量或枚举。
6. Java 代码避免复杂表达式链（多层三元、长链 Stream、嵌套表达式），优先显式 `if/for`。
7. 单个方法建议不超过 60 行，超过必须拆分私有方法并补充方法注释。

### 2.2 部署约束

1. 完全内网部署，数据不出企业网络
2. 全链路HTTPS/WSS
3. 核心操作可审计（谁在何时做了什么）
4. 开发环境固定：`Windows 11`
5. 生产环境固定：`CentOS 7`

## 三、钉钉对标能力地图（功能）

说明：本章节定义“对标目标能力”，用于研发对齐，不等同于逐像素复刻。

### 3.1 P0：即时沟通核心能力（必须对标）

1. 会话体系：单聊、群聊、置顶、免打扰、未读管理
2. 消息能力：文本、图片、文件、语音、视频、引用回复、转发、撤回、编辑、@提及、表情回应
3. 消息状态：发送中、发送失败、重试、已读/未读、回执查看
4. 搜索能力：会话搜索、消息搜索、成员搜索
5. 上传能力：点击上传、拖拽上传、粘贴上传、进度反馈、失败重试

### 3.2 P1：高频协作能力（应完整实现）

1. DING强提醒（发送、确认、回执）
2. 群能力：公告、群文件、成员管理、管理员、禁言
3. 工作台高频：审批、考勤、待办、日程、工作报告
4. 通话能力：语音通话、视频通话（WebRTC实链路）

### 3.3 P2：增强能力（可分阶段）

1. 链接卡片预览
2. 位置消息、名片消息
3. 群待办、群投票
4. 应用中心与报表扩展

## 四、钉钉对标UI布局规范（Web，钉钉主对标）

本章是UI改造的硬约束，不满足视为未完成。

### 4.1 总体信息架构

桌面端采用三段式结构：

1. 左侧主导航：模块切换（消息、通讯录、工作台等）
2. 中左会话栏：搜索 + 会话列表 + 快捷动作
3. 右侧主内容区：聊天区（头部、消息流、输入区）+ 可切换详情区

### 4.2 布局尺寸规范（基准值）

1. 主导航宽度：`68px`
2. 会话栏宽度：`320px`（允许响应式 300-340）
3. 聊天主区最小宽度：`720px`
4. 右侧详情栏宽度：`320px`（抽屉/常驻二选一）
5. 统一8pt栅格：间距只允许 `4/8/12/16/20/24/32`

### 4.3 视觉层级规范

1. 字号层级：
   - 一级标题：18
   - 会话名：15
   - 正文消息：14
   - 辅助信息：12
   - 次级标注：11
2. 圆角体系：
   - 卡片：12/16
   - 按钮：6/8
   - 气泡：对方 `2 12 12 2`，自己 `12 12 2 12`
3. 状态色：
   - 主色：#1677FF
   - 成功：#52C41A
   - 警告：#FAAD14
   - 危险：#FF4D4F
4. 不允许大面积“空白灰屏”，无数据必须有可行动空态。

### 4.4 页面级规范

#### 4.4.1 主页面（MainPage）

1. `chat`模块下，聊天面板必须始终渲染
2. 无会话时显示欢迎态（含快捷引导），禁止纯空白
3. 侧边导航、会话栏、聊天区分割线清晰

#### 4.4.2 会话栏（SessionPanel）

1. 顶部必须有：标题、快捷创建、搜索
2. 会话项必须有：头像、名称、时间、预览、未读、置顶/免打扰状态
3. 排序规则：置顶 > 未读 > 最近时间
4. 空态必须可直接引导“发起会话”

#### 4.4.3 聊天区（ChatPanel）

1. 顶部：会话信息 + 通话 + 搜索 + 更多
2. 中部：消息流（时间分割、未读定位、引用跳转）
3. 底部：输入区（多行输入、上传、表情、@、发送快捷键）
4. 所有按钮禁止“开发中”死路，必须可达或隐藏

### 4.5 组件级规范（关键）

#### 4.5.1 MessageBubble

1. 模板与样式类必须一致（`.bubble`必须实际挂载）
2. 悬停操作条：回复、转发、收藏、多选、撤回/删除
3. 支持URL识别与点击
4. 文件消息显示名称、大小、下载行为
5. 消息状态和气泡关系明确（发送中/失败）

#### 4.5.2 MessageInput

1. 支持拖拽、粘贴、点击上传三路径
2. 上传反馈必须含进度和失败重试
3. 群聊支持`@成员`快速插入
4. 输入状态同步（typing）可见且可自动消退

#### 4.5.3 ChatHeader

1. 语音/视频按钮必须走真实业务事件
2. “查看文件/搜索/置顶/免打扰/清空”必须有可执行动作
3. 单聊与群聊头部信息展示差异化

### 4.6 二级菜单信息架构（必须冻结执行）

说明：一级导航点入后，必须进入对应的二级菜单页签，不允许“单页承载所有入口”。

#### 4.6.1 一级：消息（Chat）

二级菜单固定为：

1. 最近会话（默认）
2. 未读
3. 置顶
4. 免打扰
5. 群聊
6. 文件与链接

#### 4.6.2 一级：通讯录（Contacts）

二级菜单固定为：

1. 组织架构
2. 我的好友
3. 群组
4. 新的联系人（申请/通知）
5. 常用联系人

#### 4.6.3 一级：工作台（Workbench）

二级菜单固定为：

1. 常用应用
2. 审批
3. 考勤
4. 待办
5. 日程
6. 工作报告

### 4.7 三级弹窗规范矩阵（高频弹窗必须统一）

说明：三级弹窗指“从二级页面触发的业务弹窗”，需统一标题区、操作区、关闭规则。

| 一级 | 二级菜单 | 三级弹窗（必须） | 核心动作 |
|------|------|------|------|
| 消息 | 最近会话/群聊 | 新建会话弹窗 | 单聊/群聊创建、成员选择 |
| 消息 | 最近会话 | 会话详情弹窗 | 置顶、免打扰、清空未读、删除会话 |
| 消息 | 文件与链接 | 文件预览弹窗 | 预览、下载、转发 |
| 消息 | 任意会话 | 发起通话弹窗 | 语音/视频邀请、设备检测、取消 |
| 通讯录 | 组织架构 | 成员详情弹窗 | 发消息、发DING、保存联系人 |
| 通讯录 | 新的联系人 | 好友申请处理弹窗 | 同意、拒绝、备注 |
| 通讯录 | 群组 | 群设置弹窗 | 群公告、成员管理、群禁言 |
| 工作台 | 审批 | 发起审批弹窗 | 模板选择、字段填写、提交 |
| 工作台 | 考勤 | 补卡申请弹窗 | 时间选择、原因、提交 |
| 工作台 | 待办 | 新建待办弹窗 | 标题、截止时间、负责人 |
| 工作台 | 日程 | 新建日程弹窗 | 时间、参会人、提醒 |
| 工作台 | 工作报告 | 新建报告弹窗 | 内容、附件、抄送 |

弹窗交互硬约束：

1. 所有三级弹窗必须支持 ESC 关闭、右上角关闭、取消按钮关闭。
2. 提交按钮必须有 loading、禁重入、防重复提交。
3. 必填项校验失败必须有字段级错误提示。
4. 破坏性动作（删除、清空）必须二次确认。
5. 弹窗关闭后，列表数据必须按需刷新，不允许“看起来成功但列表没变化”。

## 五、功能优先级与完成定义

### 5.1 优先级定义

| 优先级 | 定义 | 标准 |
|--------|------|------|
| P0 | 核心必达 | 可用、稳定、可测、无死路 |
| P1 | 重要能力 | 完整流程闭环 |
| P2 | 增强能力 | 可延后，但需预留架构 |

### 5.2 当前阶段重点（2026-03）

#### P0（立即完成）

1. 语音消息（录制、上传、播放、进度）
2. 语音/视频通话接入WebRTC
3. 消息收藏全链路
4. 右侧详情栏（群信息、成员、文件、公告）
5. 输入上传链路稳定化（拖拽/粘贴/进度）

#### P1（本季度）

1. 群文件前端完善（分类、预览、权限）
2. 链接卡片自动识别与预览
3. 群待办与群投票
4. 通讯录前端完善（好友申请、分组）

### 5.3 Java + Vue 双端配对开发原则（强制）

为避免“只做UI、后端缺失”或“只写后端、前端不可用”，所有 P0/P1 功能必须按双端配对交付。

1. 每个功能至少包含两个任务：
   - `WEB-*`：Vue3 页面/组件/状态管理/API调用
   - `API-*`：Java 接口/Service/Mapper/SQL/权限校验
2. WEB 任务 AC 不允许只写“按钮可点击”，必须包含接口成功/失败反馈。
3. API 任务 AC 不允许只写“接口可返回”，必须包含业务校验和错误码。
4. 联调通过后才允许标记 done：
   - 前端 `npm run build`
   - 后端 `mvn -DskipTests compile`
5. 任何聊天核心能力（发送、已读、置顶、免打扰、搜索）必须先有 API 再做 UI 行为。
6. Java 代码必须遵守：
   - JDK8 语法
   - 禁止魔法值
   - 禁止复杂表达式链
   - 优先可维护性、可靠性、可用性

## 六、性能与稳定性要求

### 6.1 性能指标

1. 在线并发：100（峰值500）
2. 消息延迟：P95 <= 300ms（内网）
3. 首屏加载：<= 3s
4. 文件上传：单文件 <= 100MB

### 6.2 稳定性指标

1. WebSocket断线重连成功率 >= 99%
2. 发送失败重试成功率 >= 95%
3. 核心页面白屏率 < 0.1%

---

## 七、安全与合规要求

1. 全站强制鉴权（JWT）
2. RBAC权限边界明确
3. 敏感词过滤可配置
4. 操作日志可追踪（删除、撤回、权限变更）
5. 消息加密策略落地（AES-256-GCM）

## 八、Claude大模型全栈开发约束（强制）

本章用于约束你当前使用的Claude，避免“看起来做了、实际上不可用”。

### 8.1 统一目标约束

Claude每次任务必须同时满足：

1. 功能可用：不允许只做UI壳
2. 代码可编译：必须通过构建
3. 流程可闭环：不允许“开发中”占位交付
4. 变更可追踪：必须给出改动文件与原因

### 8.2 禁止行为（红线）

1. 禁止把未实现功能伪装成已完成
2. 禁止仅改文案不打通事件
3. 禁止引入与需求无关的大面积重构
4. 禁止输出“理论方案”但不落地代码
5. 禁止用Mock覆盖真实API问题而不标注

### 8.3 输出格式约束（每次任务）

Claude输出必须包含以下5段：

1. 任务理解（1-3句）
2. 改动清单（文件路径 + 作用）
3. 关键实现说明（为什么这么做）
4. 验证结果（build/test/lint）
5. 风险与下一步（最多3条）

### 8.4 工程执行约束

1. 每次最多聚焦一个主目标（例如“会话栏重构”）
2. UI改动必须同步状态与交互逻辑
3. 新增组件必须接入主流程，不允许孤立文件
4. 任务结束必须跑`npm run build`
5. 关键路径必须至少手工点测：
   - 选会话

## 十七、LLM规划开发协议（可直接执行）

本章节定义“后期由大模型直接规划开发”的统一协议。  
任何模型（Claude/Codex/其他）执行本项目任务，必须遵守。

### 17.1 任务输入标准（Definition of Ready）

每个任务在开始前必须具备以下字段：

1. `task_id`：唯一编号（如 `IM-A-001`）
2. `goal`：一句话目标（可验证）
3. `scope_in`：允许改动文件/模块
4. `scope_out`：禁止改动模块
5. `acceptance`：可执行验收条目（至少3条）
6. `verify_cmd`：验证命令（至少包含 `npm run build`）
7. `risk`：风险级别（low/medium/high）

缺任何字段，模型不得开工。

### 17.2 任务完成标准（Definition of Done）

任务完成必须同时满足：

1. 代码已提交到工作区（非仅方案）
2. `verify_cmd`全部执行且通过
3. 验收条目逐条给出结果
4. 列出变更文件绝对路径
5. 明确“未完成项/后续项”（可为空，但必须显式写“无”）

模型做规划时，输出必须是以下结构：

```text
[任务编号] IM-A-001

[范围-包含] ...
[范围-排除] ...
[实现步骤]
1. ...
2. ...
3. ...
[验收标准]
1. ...
2. ...
3. ...
[验证命令]
1. npm run build
2. ...
[风险与回滚]
- 风险:
- 回滚:
```

### 17.4 开发输出格式（强制模板）


### 23.1 可达性指标

1. 发消息主入口：1次点击可达
2. 上传入口：2次点击内可达
3. 常用操作（回复/转发/收藏）：悬停后1次点击可达
4. 通话入口（语音/视频）：头部和用户详情都可达

### 23.2 反馈指标

1. 消息发送状态必须可见（发送中/成功/失败）
2. 上传必须可见进度和失败重试
3. 通话状态必须可见（邀请/接听/拒绝/挂断）
4. 网络异常必须有提示与降级行为

### 23.3 阅读效率指标

1. 时间分割语义化（今天/昨天/日期）
2. 消息气泡层级清晰（自己/对方）
3. 引用消息可定位回跳
4. 链接至少可点击，优先卡片化

### 23.4 视觉一致性指标

1. 仅允许统一字号层级（18/15/14/12/11）
2. 仅允许统一间距层级（4/8/12/16/20/24/32）
3. 暗色模式必须保持可读对比度
4. 禁止无意义装饰造成信息密度下降

### 23.5 禁止项

1. 禁止“开发中”文案替代功能交付
2. 禁止只改视觉不改交互
3. 禁止改完后不跑构建验证

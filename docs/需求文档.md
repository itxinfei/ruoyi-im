## 一、项目概述

### 1.1 项目背景

**核心诉求：数据安全，内网部署**

随着企业数字化转型的深入，即时通讯已成为企业协同办公的基础设施。然而，使用第三方公共IM服务（如钉钉、企业微信等）存在以下**数据安全隐患**：

| 风险类型 | 具体描述 |
|----------|----------|
| **数据外泄风险** | 聊天记录、文件传输等企业核心数据存储在第三方服务器 |
| **不可控风险** | 无法控制数据的访问权限、存储位置、销毁方式 |
| **合规风险** | 某些行业（金融、政府、军工）禁止数据外传 |
| **依赖风险** | 依赖外部服务，服务中断影响业务连续性 |

**解决方案**：本项目开发一个**Web版钉钉5.6的完整复刻版本**，部署在企业**内网服务器**上，所有数据（聊天记录、文件、用户信息等）完全存储在企业自有服务器中，从根本上杜绝数据外泄风险。

### 1.2 项目目标

**100%复刻钉钉PC 5.6版本**，在浏览器中实现原生应用般的体验：

| 维度 | 目标描述 |
|------|----------|
| **UI一致性** | 像素级复刻钉钉5.6的界面布局、颜色、字体、图标、动画效果 |
| **功能完整性** | 消息、DING、工作台、通讯录、文档、应用中心、邮箱全覆盖 |
| **交互一致性** | 完全复刻钉钉5.6的操作逻辑、快捷键、右键菜单、拖拽交互 |
| **业务逻辑一致性** | 消息流转、审批流程、考勤规则等业务逻辑与钉钉5.6保持一致 |
| **数据安全性** | **内网部署，数据不经过任何第三方服务器，完全自主管控** |

## 二、产品定位与核心特性

### 2.1 产品定义

面向中大型企业的私有化即时通讯与协同办公平台，支持本地服务器/私有云部署，提供组织管理、即时通讯、工作台、审批流转、应用聚合等核心能力，实现企业数据全量自主管控。

### 2.2 核心特性

#### 2.2.1 私有化部署
- 支持本地服务器/私有云部署
- 数据全量存储于企业自有服务器
- 支持部署环境隔离（开发/测试/生产）
- 生产环境可关闭外部网络访问

#### 2.2.2 完整IM能力
- 单聊、群聊消息收发
- 消息回执与已读状态
- 文件/图片传输
- 消息撤回/编辑
- 聊天记录查询

#### 2.2.3 灵活组织架构
- 复用若依sys_dept/sys_user实现企业-部门-成员层级管理
- 支持自定义组织架构配置
- 支持组织架构同步与导入

#### 2.2.4 协同办公能力
- 可定制化工作台
- 审批流程引擎
- 应用中心聚合

### 2.3 适用场景

| 场景 | 描述 |
|------|------|
| 数据敏感型企业 | 金融、政府、科研等对数据安全要求高的行业 |
| 内部协同需求 | 需要统一通讯与办公平台的中大型企业 |
| 系统集成需求 | 需要将通讯能力与企业现有系统集成的场景 |

#### 4.2.6 多端同步、离线消息与漫游规则

> 目标：PC Web 与移动端、其他 PC 端保持消息同步体验，用户换设备不会丢消息。

| 功能点 | 钉钉5.6表现 | Web实现要求 |

| 离线消息 | 下线期间收到的消息，上线后自动同步并按时间插入 | 支持按会话增量拉取，避免一次性全量加载；按发送时间排序 |

#### 4.2.7 性能与容量要求（聊天场景）

> 目标：在大群、大消息量场景下仍然保持流畅体验，性能体验不低于钉钉PC 端。

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 单会话消息量 | 支持至少 5 万条消息历史漫游 | 通过分页+虚拟列表实现，避免一次性渲染 |
| 首屏加载时间 | 在最近 200 条消息内，首屏渲染时间 ≤ 500ms | 不包含首个接口响应时间，仅指前端渲染 |
| 会话切换延迟 | 切换会话到可操作时间 ≤ 300ms | 包含历史消息加载和 DOM 渲染 |
| WebSocket 重连 | 网络恢复后 5 秒内完成重连 | 超时需提示用户手动刷新 |

前端性能要求：
- 必须使用虚拟滚动或按需渲染技术渲染消息列表；
- 图片、文件缩略图使用懒加载，滚动到可视区域才加载；
- 任何单次 DOM 更新不应阻塞主线程超过 16ms。

| 退群 / 踢人 | 成员可主动退出群聊；群主/管理员可移除成员 | 退群/踢人需产生系统消息（包括操作人、时间、被操作人）；退群后不再收到新消息，但保留历史记录 |
| 群公告 | 群主/管理员可发布公告，在聊天顶部或侧边展示 | 公告内容支持富文本（标题+正文+附件）；新公告在消息区以系统消息形式提醒，右侧详情面板可查看历史公告 |
| 群文件 | 群内上传的文件可在“群文件”列表中统一管理 | 群文件列表支持按上传人、时间、类型过滤；与文件消息复用同一存储；支持按权限删除文件 |
| @成员 | 支持 @单人/@全体成员，提醒效果突出 | 输入 @ 时弹出成员选择列表；@消息在消息列表中以特殊高亮展示；被 @ 成员未读列表中有醒目标记 |
| 群名片 | 在联系人或搜索结果中展示群名片（名称、成员数量、群类型） | 群名片点击进入群聊或右侧详情面板；样式对齐钉钉群名片布局 |
| 群搜索 | 支持按群名搜索群聊 | 在全局搜索和会话搜索中均可按群名检索；匹配规则与钉钉相同（前缀、模糊） |

权限与安全要求：
- 群主具备最高权限，可转让群主、解散群聊、设置管理员；
- 管理员权限：审核邀请、管理群公告、管理群文件、禁言（后续可扩展）；
- 禁言规则（预留）：支持全员禁言/部分禁言；被禁言成员发送消息时提示“当前群聊已被禁言”。

#### 4.2.10 OA卡片消息（审批 / 考勤 / 日程等在IM中的呈现）

> 目标：将审批、考勤、日程等 OA 事件以“消息卡片”的形式融入 IM，会话成为一切业务的入口，与钉钉体验一致。

| 卡片类型 | 钉钉5.6表现 | Web实现要求 |
|----------|------------|------------|
| 审批卡片 | 在消息中展示审批标题、发起人、当前状态、时间等，点击进入审批详情 | 统一的审批卡片样式：左侧图标、右侧标题+状态+简要信息；点击卡片在右侧详情面板或新页签打开审批详情；卡片支持"通过/驳回"快捷按钮（视权限而定） |
| 考勤卡片 | 展示打卡结果（正常/迟到/缺卡）、时间、地点 | 打卡异常自动推送卡片到“系统通知/考勤助手”会话；卡片点击查看当日考勤详情；支持对迟到/缺卡发起补卡申请 |
| 日程卡片 | 展示会议/日程标题、时间、参与人、地点 | 日程创建/变更/取消时推送到相关参与者会话；卡片上提供“查看详情”“加入会议”等按钮；支持与视频会议模块联动 |
| 任务卡片 | 展示任务标题、截止时间、责任人、状态 | 任务创建或状态变更时推送；卡片支持“标记完成”按钮；状态变化同步到任务模块 |
| 系统通知卡片 | 展示系统公告、配置变更、安全提示等 | 风格统一为灰底系统卡片；点击查看详情页面；支持“稍后不再提示”等操作（可选） |

交互与状态要求：
- 卡片消息本质上是一条消息，具备消息ID、时间戳、发送方（系统/应用）等基础字段；
- 卡片需支持“展开/收起”模式，默认展示关键信息，展开后显示更多字段；
- 卡片上的操作（审批通过、拒绝、标记完成等）必须有明确的状态反馈，如按钮变成“已处理”、卡片状态标签变更。

内网部署要求：
- 所有卡片数据来源于内网服务（ruoyi-im-api + 业务模块），不得访问外网 API；
- 卡片中若需展示图标/缩略图，使用本地静态资源或内网图床；
- 提供统一的“卡片消息类型字段”和“业务关联ID字段”，便于后端审计与追踪。

#### 10.1.2 消息类型定义

| 消息类型 | Type值 | 说明 | 优先级 |
|----------|--------|------|--------|
| CHAT | chat | 普通聊天消息 | 高 |
| SYSTEM | system | 系统通知消息 | 高 |

| TYPING | typing | 对方正在输入 | 低 |
| READ_ACK | read | 消息已读回执 | 高 |
| HEARTBEAT | heartbeat | 心跳保活消息 | 最高 |
| ERROR | error | 错误消息 | 高 |

#### 10.1.3 心跳保活机制

| 参数 | 值 | 说明 |
|------|-----|------|
| 心跳间隔 | 30秒 | 客户端与服务器之间的心跳检测间隔 |
| 心跳超时 | 60秒 | 超过60秒未收到心跳视为连接断开 |
| 重连策略 | 指数退避 | 首次重连等待1秒，后续每次翻倍，最长30秒 |
| 最大重连次数 | 5次 | 超过5次重连失败则提示用户 |

#### 10.1.4 连接认证流程

```
用户登录
    │
    ▼
获取JWT Token（含userId）
    │
    ▼
建立WebSocket连接
    │
    ▼
发送认证消息（AuthMessage）
    │  {token: "xxx", userId: 123}
    │
    ▼
服务器验证Token
    │
    ├── 验证通过 ──→ 建立Channel与UserId映射 ──→ 加入用户Channel组
    │
    └── 验证失败 ──→ 关闭连接，返回错误码
```

#### 10.1.5 消息推送策略

| 场景 | 推送方式 | 实现说明 |
|------|----------|----------|
| 单聊消息 | 单点推送 | 根据receiverId查询Channel，直接推送 |
| 群聊消息 | 广播推送 | 根据groupId查询所有成员Channel，批量推送 |
| 系统通知 | 广播推送 | 推送给所有在线用户 |
| 用户上线 | 广播推送 | 通知好友/群组成员 |
| 用户离线 | 延迟推送 | 消息存入Redis，离线用户上线后拉取 |

```
│  │  - 按月分表（im_message_202501, im_message_202502...）   │   │
│  │  - 冷热数据分离（热数据MySQL，冷数据归档到历史库）        │   │
│  │  - 消息清理策略（超过N天自动清理或归档）                  │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

#### 10.2.3 消息存储流程

```
发送消息
    │
    ▼
1. 消息预处理
   - 消息加密（AES加密）
   - 生成clientMsgId（客户端消息ID，用于幂等校验）
   - 验证发送权限
    │
    ▼
2. 消息持久化
   - 插入im_message表
   - 更新im_conversation表的last_message_id和last_message_time
   - 更新im_conversation表的unread_count（+1）
    │
    ├── 在线用户 ──→ 3a. WebSocket推送
    │
    └── 离线用户 ──→ 3b. 存入Redis离线消息队列
                       （zadd offline:userId message score）
    │
    ▼
4. 返回发送结果
   - 返回messageId给发送方
   - 标记消息为已发送状态
```

#### 10.2.4 离线消息处理

| 处理阶段 | 实现方式 | 说明 |
|----------|----------|------|
| 消息缓存 | Redis ZSET | 按时间戳排序，支持分页拉取 |
| 未读计数 | Redis INCR/DECR | 原子操作，保证计数准确 |
| 消息拉取 | 分页获取 | 上线时拉取最近N条离线消息 |
| 计数同步 | 实时更新 | 用户上线后同步未读计数到前端 |
| 消息清理 | 定时任务 | 超过7天未领取的离线消息自动清理 |

#### 10.2.5 消息分表策略

| 策略 | 说明 | 适用场景 |

## 十一、安全增强设计

### 11.1 数据加密体系

#### 11.1.1 传输加密

| 层级 | 加密方式 | 实现说明 |
|------|----------|----------|
| 传输层 | TLS 1.2/1.3 | 生产环境强制HTTPS/WSS |
| 应用层 | AES-256-GCM | 消息内容端到端加密 |
| 密钥交换 | RSA-2048 | 传输对称密钥 |
| 签名机制 | HMAC-SHA256 | 消息完整性校验 |

#### 11.1.2 存储加密

| 数据类型 | 加密方式 | 说明 |
|----------|----------|------|
| 消息内容 | AES-256 | 密文存储，支持按需解密 |
| 用户密码 | BCrypt | 哈希加盐，不可逆 |
| Token密钥 | AES-256 | 密文存储 |
| 文件加密 | AES-256 | 大文件分块加密 |
| 敏感字段 | 字段级加密 | 手机号、身份证等 |

#### 11.1.3 端到端加密流程

```
发送方
    │
    ▼
1. 生成随机对称密钥（Session Key）
    │
    ▼
2. 用Session Key加密消息内容（AES）
    │
    ▼
3. 用接收方公钥加密Session Key（RSA）
    │
    ▼
4. 发送加密数据 + 加密的Session Key
    │
    ────────────────────────────→ 网络传输
    │
接收方
    │
    ▼
1. 用私钥解密Session Key（RSA）
    │
    ▼
2. 用Session Key解密消息内容（AES）
```

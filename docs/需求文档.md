## 一、项目目标与范围

### 1.1 产品目标

1. 在企业内网场景下，提供稳定、低延迟、高可用的即时通讯能力。
2. 在主流程体验上达到“钉钉可替代级”：
   - 找人快
   - 发消息快
   - 看消息清晰
   - 协作入口不绕路
3. 工作台模块达到“可日常使用级”，不是只展示。

### 1.2 目标用户

1. 10-500人规模企业员工（内网办公）
2. 组织管理员（权限、组织、审计）
3. 部门负责人（群组、审批、待办协作）

### 1.3 非目标（当前版本不追求）

1. 公网社交属性
2. 开放生态应用市场完整复刻
3. 超大规模万人并发架构

## 二、技术与部署基线

### 2.1 技术栈

- 前端：Vue 3 + Vite + Element Plus + Vuex + WebSocket
- 后端：Spring Boot + Spring Security + JWT + MyBatis Plus
- 存储：MySQL + Redis + 本地文件系统

### 2.2 部署约束

1. 完全内网部署，数据不出企业网络
2. 全链路HTTPS/WSS
3. 核心操作可审计（谁在何时做了什么）

## 三、钉钉对标能力地图（功能）

说明：本章节定义“对标目标能力”，用于研发对齐，不等同于逐像素复刻。

### 3.1 P0：即时沟通核心能力（必须对标）

1. 会话体系：单聊、群聊、置顶、免打扰、未读管理
2. 消息能力：文本、图片、文件、语音、视频、引用回复、转发、撤回、编辑、@提及、表情回应
3. 消息状态：发送中、发送失败、重试、已读/未读、回执查看
4. 搜索能力：会话搜索、消息搜索、成员搜索
5. 上传能力：点击上传、拖拽上传、粘贴上传、进度反馈、失败重试

### 3.2 P1：高频协作能力（应完整实现）

1. DING强提醒（发送、确认、回执）
2. 群能力：公告、群文件、成员管理、管理员、禁言
3. 工作台高频：审批、考勤、待办、日程、工作报告
4. 通话能力：语音通话、视频通话（WebRTC实链路）

### 3.3 P2：增强能力（可分阶段）

1. 链接卡片预览
2. 位置消息、名片消息
3. 群待办、群投票
4. 应用中心与报表扩展

## 四、钉钉对标UI布局规范（Web）

本章是UI改造的硬约束，不满足视为未完成。

### 4.1 总体信息架构

桌面端采用三段式结构：

1. 左侧主导航：模块切换（消息、通讯录、工作台等）
2. 中左会话栏：搜索 + 会话列表 + 快捷动作
3. 右侧主内容区：聊天区（头部、消息流、输入区）+ 可切换详情区

### 4.2 布局尺寸规范（基准值）

1. 主导航宽度：`68px`
2. 会话栏宽度：`320px`（允许响应式 300-340）
3. 聊天主区最小宽度：`720px`
4. 右侧详情栏宽度：`320px`（抽屉/常驻二选一）
5. 统一8pt栅格：间距只允许 `4/8/12/16/20/24/32`

### 4.3 视觉层级规范

1. 字号层级：
   - 一级标题：18
   - 会话名：15
   - 正文消息：14
   - 辅助信息：12
   - 次级标注：11
2. 圆角体系：
   - 卡片：12/16
   - 按钮：6/8
   - 气泡：对方 `2 12 12 2`，自己 `12 12 2 12`
3. 状态色：
   - 主色：#1677FF
   - 成功：#52C41A
   - 警告：#FAAD14
   - 危险：#FF4D4F
4. 不允许大面积“空白灰屏”，无数据必须有可行动空态。

### 4.4 页面级规范

#### 4.4.1 主页面（MainPage）

1. `chat`模块下，聊天面板必须始终渲染
2. 无会话时显示欢迎态（含快捷引导），禁止纯空白
3. 侧边导航、会话栏、聊天区分割线清晰

#### 4.4.2 会话栏（SessionPanel）

1. 顶部必须有：标题、快捷创建、搜索
2. 会话项必须有：头像、名称、时间、预览、未读、置顶/免打扰状态
3. 排序规则：置顶 > 未读 > 最近时间
4. 空态必须可直接引导“发起会话”

#### 4.4.3 聊天区（ChatPanel）

1. 顶部：会话信息 + 通话 + 搜索 + 更多
2. 中部：消息流（时间分割、未读定位、引用跳转）
3. 底部：输入区（多行输入、上传、表情、@、发送快捷键）
4. 所有按钮禁止“开发中”死路，必须可达或隐藏

### 4.5 组件级规范（关键）

#### 4.5.1 MessageBubble

1. 模板与样式类必须一致（`.bubble`必须实际挂载）
2. 悬停操作条：回复、转发、收藏、多选、撤回/删除
3. 支持URL识别与点击
4. 文件消息显示名称、大小、下载行为
5. 消息状态和气泡关系明确（发送中/失败）

#### 4.5.2 MessageInput

1. 支持拖拽、粘贴、点击上传三路径
2. 上传反馈必须含进度和失败重试
3. 群聊支持`@成员`快速插入
4. 输入状态同步（typing）可见且可自动消退

#### 4.5.3 ChatHeader

1. 语音/视频按钮必须走真实业务事件
2. “查看文件/搜索/置顶/免打扰/清空”必须有可执行动作
3. 单聊与群聊头部信息展示差异化

## 五、功能优先级与完成定义

### 5.1 优先级定义

| 优先级 | 定义 | 标准 |

3. 消息收藏全链路
4. 右侧详情栏（群信息、成员、文件、公告）
5. 输入上传链路稳定化（拖拽/粘贴/进度）

#### P1（本季度）

1. 群文件前端完善（分类、预览、权限）
2. 链接卡片自动识别与预览
3. 群待办与群投票
4. 通讯录前端完善（好友申请、分组）

## 六、性能与稳定性要求

### 6.1 性能指标

1. 在线并发：100（峰值500）
2. 消息延迟：P95 <= 300ms（内网）
3. 首屏加载：<= 3s
4. 文件上传：单文件 <= 100MB

### 6.2 稳定性指标

1. WebSocket断线重连成功率 >= 99%
2. 发送失败重试成功率 >= 95%
3. 核心页面白屏率 < 0.1%

---

## 七、安全与合规要求

1. 全站强制鉴权（JWT）
2. RBAC权限边界明确
3. 敏感词过滤可配置
4. 操作日志可追踪（删除、撤回、权限变更）
5. 消息加密策略落地（AES-256-GCM）

## 八、Claude大模型全栈开发约束（强制）

本章用于约束你当前使用的Claude，避免“看起来做了、实际上不可用”。

### 8.1 统一目标约束

Claude每次任务必须同时满足：

1. 功能可用：不允许只做UI壳
2. 代码可编译：必须通过构建
3. 流程可闭环：不允许“开发中”占位交付
4. 变更可追踪：必须给出改动文件与原因

### 8.2 禁止行为（红线）

1. 禁止把未实现功能伪装成已完成
2. 禁止仅改文案不打通事件
3. 禁止引入与需求无关的大面积重构
4. 禁止输出“理论方案”但不落地代码
5. 禁止用Mock覆盖真实API问题而不标注

### 8.3 输出格式约束（每次任务）

Claude输出必须包含以下5段：

1. 任务理解（1-3句）
2. 改动清单（文件路径 + 作用）
3. 关键实现说明（为什么这么做）
4. 验证结果（build/test/lint）
5. 风险与下一步（最多3条）

### 8.4 工程执行约束

1. 每次最多聚焦一个主目标（例如“会话栏重构”）
2. UI改动必须同步状态与交互逻辑
3. 新增组件必须接入主流程，不允许孤立文件
4. 任务结束必须跑`npm run build`
5. 关键路径必须至少手工点测：
   - 选会话
   - 发消息
   - 上传文件
   - 右键菜单
   - 头部操作按钮

### 8.5 质量闸门（DoD）

满足以下条件才允许标记“完成”：

1. 编译通过
2. 无明显UI错位（重叠、溢出、空白大块）
3. 无“开发中”死路按钮
4. 无控制台高频报错
5. 文档与代码状态一致

## 十、研发流程（AI协同版）

### 10.1 推荐迭代节奏

1. 需求切片：按页面主流程拆分（每片0.5-1天）
2. Claude实现：先结构，再交互，再视觉微调
3. 人工验收：功能闭环 + 视觉一致性
4. 回写文档：同步实际状态

### 10.2 任务切片建议

1. 切片A：MainPage + Chat空态
2. 切片B：SessionPanel结构与排序
3. 切片C：MessageBubble与MessageItem统一
4. 切片D：ChatHeader事件打通
5. 切片E：输入区上传链路与typing

## 十一、验收清单（产品+设计+开发+测试）

### 11.1 产品验收

1. 主路径无断点（找人-发消息-看回执）
2. 关键入口无“开发中”提示
3. 空态可引导下一步动作

### 11.2 设计验收

1. 三段式布局稳定
2. 字号、间距、颜色符合规范
3. 暗色模式无明显对比问题

### 11.3 开发验收

1. 编译通过
2. 无阻断报错
3. 新增逻辑有降级处理

### 11.4 测试验收

1. 消息收发稳定
2. 上传、撤回、编辑、转发可用
3. 会话排序、未读、置顶状态正确

---

## 十二、风险与应对

1. 风险：大模型一次性改动过多引发回归  
   应对：严格“一次一目标”，每次强制构建验证

2. 风险：UI精修与业务联调节奏冲突  
   应对：先主流程闭环，再做视觉精修

3. 风险：文档与代码再次漂移  
   应对：每次合并后更新本文档版本与状态

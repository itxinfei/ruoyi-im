# UI对标设计规范（钉钉主对标版）

> 主对标：钉钉（8.2系列）
> 次对标：飞书（信息层级与协作入口）、野火IM（IM与RTC能力完整性）
> 目标：让 RuoYi-IM 的 Web 界面在“聊天实用性”上达到企业可替代级。

---

## 1. 对标优先级

1. 第一优先：钉钉聊天主路径体验
2. 第二优先：飞书的信息层级与工作台入口组织
3. 第三优先：野火IM的IM/RTC能力覆盖与私有化可控思路

禁止做“拼贴式对标”，必须保持统一视觉语言。

---

## 2. 页面结构规范（桌面Web）

### 2.1 总体布局

1. 左导航：64px（对标钉钉 8.0+ 紧凑型布局）
2. 会话栏：240px（黄金比例，flex-shrink: 0）
3. 聊天主区：Header + MessageList + MessageInput（flex: 1, min-width: 0）
4. 详情区：320px，可抽屉/侧栏

### 2.2 空间与密度

1. 使用 8pt 栅格体系 (n * 4px)
2. 全站统一圆角：10px（锁死，禁止使用 8px 或 12px）
3. 行高与字号层级稳定：18/15/14/12/11
4. 会话项高度建议 68~76
5. 气泡最大宽度建议 <= 520


### 2.3 二级菜单规范（必须实现）

一级导航进入后，必须出现二级菜单区，不允许把所有功能堆在一个页面。

1. 消息：最近会话 / 未读 / 置顶 / 免打扰 / 群聊 / 文件与链接
2. 通讯录：组织架构 / 我的好友 / 群组 / 新的联系人 / 常用联系人
3. 工作台：常用应用 / 审批 / 考勤 / 待办 / 日程 / 工作报告

### 2.4 三级弹窗规范（必须统一）

三级弹窗是从二级菜单触发的业务弹窗，需满足统一交互：

1. 必须有标题区、内容区、操作区（取消/确定）
2. 必须支持 ESC、右上角关闭、取消关闭
3. 提交态必须有 loading 且防重复提交
4. 危险操作必须二次确认
5. 关闭后父列表按需刷新

---

## 3. 聊天模块强制要求（必须全部满足）

### 3.1 可达性

1. 发消息入口必须常驻可见
2. 上传入口（图片/文件）不得隐藏在多层菜单
3. 语音/视频入口必须可触发真实行为，不得“开发中”

### 3.2 状态反馈

1. 消息状态：发送中/成功/失败/重试
2. 会话状态：未读、置顶、免打扰
3. 输入状态：对方正在输入（自动消退）
4. 通话状态：邀请/接听/拒绝/挂断

### 3.3 阅读体验

1. 时间分割语义化（今天/昨天/日期）
2. 消息气泡层级清晰（自己/对方）
3. 引用消息可跳转定位
4. URL可识别，失败时至少保持可点击

### 3.4 可靠性

1. 拖拽上传、粘贴上传可用
2. 上传失败有可见重试路径
3. 异常网络有降级提示

---

## 4. 组件级对标要点

### 4.1 SessionPanel

1. 结构：标题/加号/搜索/列表
2. 排序：置顶 > 未读 > 时间
3. 信息元素：头像、名称、时间、摘要、未读角标

### 4.2 ChatHeader

1. 显示会话名称与在线状态
2. 提供语音、视频、更多操作
3. 更多菜单动作必须有真实处理分支

### 4.3 MessageBubble

1. 模板与样式类必须一致（避免样式失效）
2. 文本、图片、文件、视频、语音、链接类型分支清晰
3. 悬停操作条包含回复/转发/收藏/多选

### 4.4 MessageInput

1. 多行输入可调整高度
2. Enter/Ctrl+Enter规则明确
3. @成员、表情、上传、语音入口统一

---

## 5. LLM执行约束（UI专项）

1. 不允许只改颜色和圆角，不改交互。
2. 不允许页面“看起来像钉钉”，但主流程断链。
3. 每次UI任务必须附带：
   - 改动前问题
   - 改动后行为
   - 验证截图点位（或可复现路径）
4. 结束必须执行 `npm run build`。
5. UI任务必须声明“一级导航 + 二级菜单 + 三级弹窗”影响范围。
6. 未覆盖二级菜单与三级弹窗的任务，不允许标记为“钉钉对齐完成”。

---

## 6. UI验收清单（QA）

1. 无会话时是否存在可行动空态
2. 会话切换后消息区是否稳定
3. 常用操作是否 2 次点击内可达
4. 是否存在“开发中”死路按钮
5. 深色模式是否可读
6. 窄屏（>=1280）是否布局不崩

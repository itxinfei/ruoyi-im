# 后台管理系统功能与页面设计需求（V1）

> 适用范围：`ruoyi-im-web/src/views/admin/*`、`ruoyi-im-api/src/main/java/com/ruoyi/im/controller/admin/*`  
> 对齐目标：在现有“用户/群组/消息/统计”基础上，形成可发布的企业管理后台。  
> 生效日期：2026-03-04

---

## 1. 文档目标

1. 明确后台管理系统 V1 的功能边界与页面标准。
2. 明确“已实现 / 待完善 / 禁止项”，避免继续出现页面有入口但链路不闭环。
3. 为前后端联动、测试验收提供统一口径。

---

## 2. 角色与权限模型

### 2.1 角色定义

1. `SUPER_ADMIN`：全量系统配置与高风险操作权限。
2. `ADMIN`：用户、群组、消息、统计的日常运营权限。
3. `AUDITOR`（预留）：审计只读权限（V1 可先不开放菜单）。

### 2.2 权限边界（V1）

1. 用户状态变更、消息删除、群组解散：`ADMIN` 可执行。
2. 角色变更、系统参数修改：仅 `SUPER_ADMIN`。
3. 数据看板、列表查询、详情查看：`ADMIN/SUPER_ADMIN`。

---

## 3. 信息架构（后台端）

### 3.1 一级导航（固定）

1. 数据概览
2. 用户管理
3. 群组管理
4. 消息管理
5. 系统配置（V1 新增补齐）

### 3.2 页面关系

1. 数据概览用于总览与异常发现，不承载编辑动作。
2. 用户/群组/消息页面为“查询 + 批量操作 + 单项详情”结构。
3. 系统配置页面承载平台参数治理（消息策略、安全策略、上传策略）。

---

## 4. 页面设计规范（后台）

### 4.1 通用布局

1. 页面结构统一：筛选区 + 数据表格区 + 批量操作区 + 分页区。
2. 每页必须有“刷新”动作；重置筛选后必须回到第一页。
3. 批量操作必须展示执行结果：成功数、失败数、失败原因摘要。

### 4.2 表格规范

1. 操作列固定右侧，保证滚动下可见。
2. 长文本默认省略并支持 tooltip。
3. 时间统一显示格式：`YYYY-MM-DD HH:mm:ss`。

### 4.3 反馈规范

1. 成功：明确对象与结果（例如“成功删除 3 条消息”）。
2. 失败：必须展示可诊断原因（权限不足、数据不存在、状态冲突）。
3. 风险操作必须二次确认（删除、解散、角色变更、参数变更）。

---

## 5. 功能需求（按页面）

### 5.1 数据概览（Dashboard）

#### 5.1.1 必须能力

1. 总用户数、活跃用户、群组数、今日消息数卡片。
2. 近 7 天消息结构统计（文本/图片/文件）。
3. 用户角色分布统计（SUPER_ADMIN/ADMIN/USER）。

#### 5.1.2 待完善

1. 增加“异常指标卡”：失败消息、封禁用户、高频投诉词。
2. 增加时间维度切换（7/15/30 天）。

### 5.2 用户管理（UserManagement）

#### 5.2.1 必须能力

1. 按关键词/角色查询。
2. 启用/禁用用户。
3. 删除用户。
4. 分页与每页条数切换。

#### 5.2.2 待完善

1. 角色变更入口（仅 SUPER_ADMIN 可见）。
2. 用户详情抽屉（最近登录、最近会话、风险操作记录）。
3. 批量启用/禁用。

### 5.3 群组管理（GroupManagement）

#### 5.3.1 必须能力

1. 群组搜索、分页查询。
2. 解散群组与批量解散。
3. 群信息编辑（名称、描述、成员上限、全员禁言）。
4. 成员列表查看与移除成员。

#### 5.3.2 待完善

1. 群公告管理入口。
2. 群文件统计（文件数、容量、近7天新增）。
3. 群风险提示（异常增长、异常活跃）。

### 5.4 消息管理（MessageManagement）

#### 5.4.1 必须能力

1. 按关键词/类型/时间范围检索消息。
2. 消息详情查看。
3. 单条删除与批量删除。

#### 5.4.2 待完善

1. 敏感词命中标记。
2. 删除原因记录（人工删除、策略删除）。
3. 审核动作（通过/拦截）预留。

### 5.5 系统配置（SystemConfig，V1 新增必做）

#### 5.5.1 配置分组

1. 消息策略：撤回时限、最大消息长度、允许消息类型。
2. 上传策略：单文件上限、允许扩展名、分片大小。
3. 安全策略：登录失败阈值、会话过期时长、密码复杂度级别。

#### 5.5.2 页面行为

1. 参数编辑后必须可回滚到上次发布值。
2. 参数保存必须记录“操作者、时间、变更前后值”。
3. 高风险参数保存需二次确认。

---

## 6. 接口与数据契约要求

1. 前端后台 API 统一使用 `src/api/admin.js`，禁止页面散落直调 URL。
2. 所有后台写操作响应统一结构：`code`、`msg`、`data`（含成功/失败明细）。
3. 批量接口 `data` 中必须包含：
   - `successCount`
   - `failedCount`
   - `failedItems`（可选，最多返回前20项）
4. 列表接口统一分页字段：`list`、`total`、`pageNum`、`pageSize`。

---

## 7. 页面视觉与交互基线

1. 后台视觉以“高信息密度、低装饰”为原则，不做聊天端风格复用。
2. 色彩语义固定：
   - 危险操作：红色
   - 警示操作：橙色
   - 普通操作：蓝色
3. 批量操作区固定在表格下方，避免滚动后丢失操作反馈。
4. 所有弹窗必须支持 ESC 关闭、右上角关闭、取消按钮关闭。

---

## 8. 验收标准（后台 V1）

1. 五个页面均可达：概览/用户/群组/消息/系统配置。
2. 每个页面至少有 1 条真实数据链路（非模拟数据）。
3. 高风险操作均有二次确认与结果反馈。
4. 权限边界有效：`ADMIN` 不可执行 `SUPER_ADMIN` 专属动作。
5. 前端 `npm run build`、后端 `mvn -DskipTests compile` 通过。

---

## 9. 实施优先级建议

1. P0：系统配置页面补齐 + 权限校验联调。
2. P0：批量操作结果结构统一（成功/失败明细）。
3. P1：用户详情抽屉、群公告管理入口、消息敏感词标记。
4. P1：后台审计日志可视化（操作人、对象、结果、耗时）。

---

## 10. 禁止项

1. 禁止“只有页面没有接口”的空壳交付。
2. 禁止“接口可用但页面没有反馈”的静默失败。
3. 禁止在后台页面保留“开发中”可点击入口。
4. 禁止越权入口仅靠前端隐藏而后端不做权限校验。

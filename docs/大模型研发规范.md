# RuoYi-IM 大模型研发规范与约束（LLM Engineering Spec & Constraints）

> **文档目的**：约束大模型（如 Claude, Codex, Gemini 等）在本项目中的理解、规划、开发、审查行为，确保大模型生成的代码具备“生产级”可用性、高审美、高可维护性。避免出现“垃圾代码”、“幻觉代码”以及“违背设计规范”的问题。
> **适用范围**：`D:/MyCode/im/ruoyi-im-web`、`D:/MyCode/im/ruoyi-im-api`
> **版本**：v6.1
> **核心指令**：项目根目录下的 `GEMINI.md` 是最高优先级指令，本规范为详细执行手册。

---

## 一、 核心底线与红线约束（禁止触碰）

1. **绝对禁飞区（No-fly Zone）**：
   - **禁止擅自修改技术栈**：后端固定 `JDK 1.8`（禁止使用 Java 9+ 的 `var`、`record`、`sealed`、`switch ->`、`"""` 文本块、`isEmpty()`、`toList()` 等语法）。前端固定 `Vue 3 (Composition API)` + `Element Plus`。
   - **禁止 Entity 直出**：后端接口入参/出参必须使用 `DTO/VO`，严禁直接返回数据库 Entity 对象。
   - **禁止硬编码色值**：前端 `.vue` 或 `.scss` 中，**禁止**出现硬编码的 hex 色值（如 `#333`、`#1677ff`），必须统一调用 `design-tokens.scss` 中定义的 CSS 变量（如 `var(--dt-text-primary)`）。
   - **禁止“开发中”占位**：大模型交付的代码必须是“真实链路”。如果功能由于依赖暂未打通，也必须通过 Mock 数据或完整的状态机（Loading -> Success/Fail）进行闭环，严禁仅放一个 `ElMessage.info("开发中")` 来敷衍。
   - **单任务单目标**：不要为了秀技而大面积重构与当前需求无关的文件，严格控制 blast radius（爆炸半径）。

2. **垃圾代码防范（Anti-Garbage Code）**：
   - **禁止重复造轮子**：在使用时间格式化、深拷贝、URL 解析等功能时，优先使用现有的 utils 工具类（如 `src/utils/`）或 Lodash/Day.js，禁止在组件内写长篇大论的纯净 JS 工具方法。
   - **清理冗余逻辑**：如果重构废弃了旧逻辑（如用新组件替代了旧组件），必须同步删除旧代码，保持代码库的极简和高内聚。

## 二、 审美与 UI 布局强制规范（对标钉钉/飞书）

本项目的 UI 目标是：**“对标钉钉的核心体验，吸取飞书的极致排版，保持界面的现代感与呼吸感。”**

### 2.1 整体信息架构（三段式黄金比例）
- **左侧主导航（模块栏）**：固定 `64px`，图标居中，字重 `300`，极致紧凑。
- **中左二级面板（会话/联系人树）**：黄金宽度 `240px`。必须加上 `flex-shrink: 0` 和 `min-width: 0`，防止被内容撑爆。
- **右侧主工作区（聊天/详情）**：占据剩余弹性空间（`flex: 1`）。必须有 `min-width: 0` 保证内部文本出现省略号（`text-overflow: ellipsis`）时不会破坏父容器宽度。

### 2.2 组件与控件美学（细节决定工业感）
- **气泡美学锁死 (Bubble Aesthetics)**：
  - 全站聊天气泡统一采用 **10px 全对称圆角**（对标钉钉 8.0+ 现代化风格）。
  - 发送方（Own）：背景色 `var(--dt-bubble-right-bg)`，字体 `var(--dt-text-primary)`。
  - 接收方（Other）：背景色 `var(--dt-bubble-left-bg)`，字体 `var(--dt-text-primary)`。
- **头像美学 (Avatar Aesthetics)**：
  - 群组头像采用**2x2或3x3聚合网格**（对齐钉钉），单人头像采用带微小圆角的方形或正圆形。
  - 无头像数据时，必须提供按用户 ID Hash 映射的彩色底底（色盘见 `DingtalkAvatar`），中间填充首字母大写。
- **阴影与质感 (Shadows & Depth)**：
  - 禁用厚重、黑粗的 `box-shadow`。全部使用 `design-tokens.scss` 中的轻量级悬浮阴影（如 `var(--dt-shadow-card-hover)`）。
- **滚动条体验 (Scrollbar)**：
  - 禁用原生粗犷滚动条，全局统一应用 `custom-scrollbar` 类，要求宽度 `6px`，透明轨道，悬停时出现浅灰色滑块。

## 三、 代码质量与可维护性约束

### 3.1 前端 Vue 3 开发标准
- **SFC 规范**：必须使用 `<script setup>` 语法糖。
- **CSS 作用域**：所有样式必须使用 `<style scoped lang="scss">`，严禁污染全局。
- **响应式控制**：区分好 `ref`（基本数据/组件引用）和 `reactive`（复杂表单对象）。避免过度监听（`watch` 滥用）。
- **模板整洁**：HTML 模板中避免过于复杂的短路运算，复杂的判断应该提取为 `computed` 属性。

### 3.2 后端 Java 开发标准 (JDK 1.8)
- **Controller 职责**：只负责参数校验、Token/User 获取、返回 `Result.success/fail`。业务逻辑必须下沉到 Service。
- **Service 职责**：负责所有业务组装、事务控制（`@Transactional`）。严禁在 Service 中直接操控 WebSocket 链接，应抽象一层 `BroadcastService` 或事件总线（EventPublisher）进行解耦。
- **异常处理**：遇到不符合业务预期的分支，直接 `throw new BusinessException("错误信息")`，由全局异常处理器统一拦截抓取，禁止到处写 `try-catch` 并返回 null。
- **日志规范**：修改数据库的关键操作、捕捉到的核心异常必须使用 `log.info` / `log.error` 打印上下文参数（如 `userId`, `conversationId`），禁止静默吞掉异常。
- **禁止魔法值**：所有状态码、业务类型必须提取为常量或枚举。

## 四、 大模型执行与输出规范

### 4.1 任务执行四步法（Think -> Act -> Verify -> Report）
1. **理解上下文**：在进行大动作之前，使用 `grep_search` 或 `read_file` 彻底查清现有相关代码，绝不能凭空捏造（防幻觉）。
2. **制定并声明策略**：想清楚要在哪里改、怎么改、受影响的面有多大。
3. **精准手术刀修改**：使用 `replace` 精确修改局部代码。
4. **自检验证**：确认修改后是否遗留未闭环的事件，并运行对应的验证命令（如 `npm run build`）。

### 4.2 禁止的回答模式
- 不要写大段无效寒暄。
- 不要重复输出未修改的文件内容。
- 不要私自推翻规范文件，如果遇到规范不合理的地方，必须向用户提出并获得同意后方可修改。

## 五、 防呆设计与审美逻辑专项协议 (Anti-Stupid Design & Aesthetic Integrity)

### 5.1 配色血统锁定 (Color Lineage)
- **色盘白名单**：全站仅允许使用 `design-tokens.scss` 中定义的变量。
- **禁止发明颜色**：AI **严禁**自行猜测或生成任何 Hex/RGB 色值。
- **透明度规范**：仅允许使用 `rgba(var(--xxx), 0.x)`，严禁使用不透明的灰色模拟透明感。

### 5.2 交互常识红线 (Common Sense UX)
- **路径闭环**：所有弹窗（Dialog）必须有明确的取消/关闭手段。所有进入路径必须有对应的返回（Back）路径。
- **操作可达性**：按钮点击区域（Hit Area）禁止小于 `24x24px`。
- **信息降噪**：禁止在一个气泡或卡片中堆砌超过 3 种字号或 2 种以上的强调色。

### 5.3 CSS 覆盖与消失守卫 (Style Protection)
- **拒绝全局污染**：严禁直接修改 `html`, `body` 或 Element Plus 的原始类名。所有修改必须包裹在组件唯一的根 Class 下。
- **BEM 强约束**：推荐使用 `__` (element) 和 `--` (modifier) 命名（如 `.chat-item__avatar--online`）。
- **层级保护 (Z-Index)**：禁止随手写 `z-index: 9999`。必须查阅 `design-tokens` 中的 Z-Index 阶梯。

## 六、 工业级 UI 视觉实现协议 (Frontend Aesthetics & Guardrails)

### 6.1 像素级栅格系统 (The 8pt Grid)
- **间距锁死**：严禁出现 `10px`, `15px` 等随机间距。所有 `margin`, `padding`, `gap` 必须符合 `n * 4` 逻辑。
- **推荐值**：极紧凑 `4px` | 紧凑 `8px` | 标准 `12px` | 呼吸感 `16px` | 模块间距 `24px+`。

### 6.2 布局“三防”红线 (Layout Robustness)
- **防撑爆**：所有 `flex: 1` 的子容器必须强制显式声明 `min-width: 0`。
- **防溢出**：主容器必须配合 `overflow: hidden` 或 `overflow-y: auto`，严禁让长内容撑破全站三段式架构。
- **防抖动**：所有具备异步加载内容的区域，必须预设高度占位或 `v-loading`。

### 6.3 交互质感锁定 (Micro-interactions)
- **动效标配**：所有点击类元素（按钮、会话项、Icon）必须包含 `transition: all 0.2s cubic-bezier(0.4, 0, 0.2, 1)`。
- **状态闭环**：AI 提交的组件必须包含 `:hover`（背景色加深/减淡）和 `:active`（轻微缩放 `transform: scale(0.98)`）。

## 七、 质量闸门 (DoD - Definition of Done)

所有大模型交付的任务必须满足：
1. **编译通过**：前端 `npm run build` 无 Error，后端 `mvn -DskipTests compile` 无 Error。
2. **逻辑闭环**：所有 `@click` 事件均有函数实现，无“空链接”或“死按钮”。
3. **视觉闭环**：圆角、间距、阴影必须符合本项目审美标准。
4. **清理闭环**：重构产生的冗余文件、变量必须同步删除。

## 八、 任务输出标准格式 (Standard Report)

Claude 每次完成任务后，必须按以下格式报告：
1. **[任务理解]**：1-3句简述目标。
2. **[改动文件]**：列出所有修改的绝对路径。
3. **[核心逻辑]**：说明为了实现 UI 质感/后端逻辑做了哪些关键处理。
4. **[验证结果]**：报告验证命令运行结果。
5. **[残留风险]**：明确哪些功能尚未闭环。

# RuoYi-IM 大模型研发规范（LLM Engineering Spec）

> 文档目的：约束 Claude/Codex 等模型在本项目中的规划、开发、审查行为，确保输出稳定、可执行、可验收。
> 适用范围：`D:/MyCode/im/ruoyi-im-web`、`D:/MyCode/im/ruoyi-im-api`。

---

## 1. 基本原则

1. 先可用，再完美：优先打通主流程闭环。
2. 单任务单目标：一次只完成一个 `task_id`。
3. 证据驱动：任何"已完成"必须有代码和验证结果。
4. 文档同步：代码变更后同步更新需求映射矩阵。
5. 技术栈强约束：不允许擅自切换 JDK/Vue 主版本。

---

## 2. 输入规范（模型开工前）

每次请求必须包含：

1. `task_id`
2. `goal`
3. `scope_in`（允许改动文件）
4. `scope_out`（禁止改动文件）
5. `acceptance`（AC1/AC2/AC3）
6. `verify_cmd`（至少 `npm run build`）

任一字段缺失：模型必须先补齐，不得直接编码。

---

## 3. 输出规范（模型交付格式）

必须按以下顺序输出：

1. 任务理解
2. 改动文件（绝对路径）
3. 关键实现点
4. 验证结果（命令 + 结论）
5. 验收对照（AC逐条 pass/fail）
6. 遗留风险

---

## 4. 质量闸门（DoD）

所有任务完成必须满足：

1. 编译通过（前端 `npm run build`）
2. 主路径无"开发中"死路入口
3. 关键事件有父子链路（emit -> 监听 -> 动作）
4. 新增功能有异常提示与降级策略
5. 文档状态同步

---

## 5. 红线（禁止行为）

1. 只改UI样式，不接业务逻辑。
2. 用文案替代真实功能交付。
3. 修改超出 `scope_in` 的文件。
4. 报告"完成"但不跑验证命令。
5. 隐藏失败项或风险。
6. 重复创建同功能文件（如 `xx-new.vue`、`xx-final.vue`）而不复用现有文件。
7. 引入未使用代码（未用 import、未调用函数、无效分支、调试日志残留）。
8. 修改技术栈基线（例如把 JDK8 改为 JDK17）而未获明确批准。

---

## 5.1 技术栈参数硬约束（必须遵守）

| 领域 | 固定参数 |
|------|------|
| Java | **JDK 1.8**（仅允许 Java 8 语法） |
| 前端框架 | **Vue 3**（Composition API） |
| 构建 | Vite（跟随当前工程） |
| Node | 18 LTS |
| 数据库 | MySQL 5.7 |
| 缓存 | Redis 6+ |
| 开发环境 | Windows 11 |
| 生产环境 | CentOS 7 |

执行规则：

1. 后端代码禁止出现 Java 9+ 语法（`var`、`record`、switch 箭头等）。
2. 前端禁止回退 Vue2 写法，不允许新增 Options API 大型页面。
3. 新增依赖前必须说明必要性和替代方案，避免技术栈漂移。
4. 所有生成代码必须与当前仓库目录结构一致，禁止平行新建"第二套工程"。

---

## 5.2 Java 编码硬规范（JDK8）

适用范围：`ruoyi-im-api` 全部 Java 代码。

1. 仅允许 Java 8 语法，禁止 Java 9+ 特性。
2. 禁止魔法值：
   - 状态码、类型码、阈值、分页默认值、重试次数必须提取为 `static final` 常量或枚举。
3. 禁止奇怪写法和过度表达式化：
   - 禁止多层三元表达式。
   - 禁止把复杂业务塞进一行表达式。
   - 禁止过长 Stream/Lambda 链替代清晰业务流程。
4. 优先可读性：
   - 业务判断优先 `if/else` 显式分支。
   - 业务循环优先 `for`，避免"技巧型写法"。
   - 方法命名必须表达业务意图，禁止 `doXxx`、`handleXxx` 泛化命名。
5. 可维护性要求：
   - 单方法建议不超过 60 行，超出需拆分私有方法。
   - 公共方法必须有 JavaDoc，说明入参、边界条件、失败行为。
6. 可靠性要求：
   - 所有外部输入必须校验空值和范围。
   - 异常禁止吞掉；必须记录关键上下文并返回可诊断错误。
   - 涉及数据一致性的写操作必须明确事务边界。
7. 可用性要求（面向接口）：
   - 失败响应必须可读（错误码 + 可理解信息）。
   - 不得返回"成功但无实际效果"的静默失败结果。

---

## 5.3 Java 代码评审清单（Reviewer必查）

1. 是否出现魔法值硬编码。
2. 是否出现多层三元或复杂表达式链。
3. 是否有超长方法且未拆分。
4. 是否存在空指针风险和未校验输入。
5. 是否存在吞异常或缺少关键日志上下文。
6. 是否存在无事务保护的数据一致性风险。
7. 是否满足"看代码即可理解业务"的可读性标准。

---

## 5.4 阿里巴巴Java开发手册要点（强制对齐）

### 5.4.1 命名风格

- **类名**：使用 `UpperCamelCase` 风格，如 `UserService`、`ImGroupMember`
- **方法名/变量名**：使用 `lowerCamelCase` 风格，如 `getUserById`、`userName`
- **常量名**：使用 `UPPER_SNAKE_CASE` 风格，如 `MAX_RETRY_COUNT`
- **包名**：全部小写，如 `com.ruoyi.im.service`
- **禁止**：
  - 禁用拼音/中英文混用命名（如 `daYin`、`userNameMingXi`）
  - 禁用下划线开头/结尾的变量名
  - 禁用 `$` 符号（除枚举值外）

### 5.4.2 常量定义

1. **不允许**魔法值直接出现在代码中，必须提取为 `static final` 常量或枚举
2. 枚举类字段必须为 `public static final`
3. 建议按业务分组定义常量类

### 5.4.3 格式规约

1. **缩进**：4个空格（禁止Tab）
2. **单行字符数**：不超过120个
3. **方法间空行**：1行
4. **if/for/while**：即使单行也必须使用 `{}`

### 5.4.4 OOP规约

1. **禁止**直接使用浮点数进行精确比较（用 `BigDecimal`）
2. **禁止**使用 `Object.equals()` 比较可能为null的字符串
3. **强制**循环体内字符串拼接必须使用 `StringBuilder`

### 5.4.5 并发处理

1. **强制**多线程共享变量必须加 `volatile` 或使用原子类
2. **强制**SimpleDateFormat 是线程不安全的，必须使用 `DateTimeFormatter`
3. **建议**使用JDK8 `java.time` 包代替 `Date`

### 5.4.6 控制语句

1. **禁止**多层嵌套（建议不超过2层）
2. **禁止**在循环体内进行数据库/网络IO操作

### 5.4.7 注释规约

1. **类必须**添加Javadoc注释
2. **公共方法必须**添加Javadoc，说明入参、返回值
3. **禁止**使用 `// TODO` 提交生产代码

### 5.4.8 其他规约

1. **强制**前端参数必须校验（@Validated/@Valid）
2. **强制**异常禁止生吞，必须记录日志或抛出
3. **强制**使用try-with-resources管理资源
4. **建议**使用 `Optional` 避免空指针

## 6. 模型角色分工

1. `Planner`：只做任务拆解与风险评估，不写代码。
2. `Builder`：按计划编码并验证。
3. `Reviewer`：只做问题审查，优先报P0/P1。

建议三段式流水：`Planner -> Builder -> Reviewer`。

---

## 7. 任务粒度标准

一个任务应满足：

1. 改动文件 <= 6
2. 主目标 <= 1
3. 可在 0.5~1 天内完成
4. AC条目 3~5 条

超过以上标准必须拆分。

补充：

1. 同类能力优先在原文件增量改造，禁止横向复制组件。
2. 任何新增文件都必须在提交说明中回答"为何不能复用现有文件"。

---

## 8. 验证命令规范

前端任务默认命令：

1. `npm run build`
2. `npm run lint`（可选但建议）

后端任务默认命令：

1. `mvn -q -DskipTests package` 或项目约定命令
2. 单测/集成测试按任务要求执行

---

## 9. 审查优先级定义

1. `P0`：阻断上线（崩溃、编译失败、主链路不可用）
2. `P1`：高优先修复（功能偏差、状态错乱、明显体验问题）
3. `P2`：优化项（可读性、性能、细节一致性）

---

## 10. 变更记录模板

每次任务完成后追加：

- task_id
- 负责人（模型/人工）
- 变更文件
- 验证结果
- 是否更新需求矩阵

---

## 11. UI专项约束（钉钉主对标）

1. UI任务必须引用 `UI对标设计规范.md`。
2. 主对标优先级固定为：钉钉 > 飞书 > 野火IM。
3. UI交付必须包含交互可达性验证，不得只给视觉调整。
4. 聊天模块任务必须覆盖：
   - 会话切换
   - 消息发送
   - 上传路径
   - 常用操作（回复/转发/收藏）
5. 禁止"新版更漂亮但更难用"的回归。
6. UI任务必须说明：
   - 一级导航所属域（消息/通讯录/工作台）
   - 二级菜单入口
   - 三级弹窗触发点与提交流程

---

## 12. 单模型直驱模式（本项目默认）

说明：后续规划与开发默认由当前模型直接完成，不再依赖其他模型二次加工。

1. 规划基线固定为：`项目总规划-钉钉对齐V1.md` + `需求文档.md`。
2. 每次任务执行顺序固定：
   - 读取文档基线
   - 仅实现一个 task_id
   - 本地验证
   - 回写文档状态
3. 禁止"多模型来回改文档"导致版本漂移。
4. 若任务与基线冲突，必须先更新文档，再实施代码。
5. 任务输出必须包含：
   - 改动文件清单
   - 关键实现说明
   - 验证命令结果
   - 风险与后续建议
6. 任何"只给建议不落地"输出视为未完成任务。

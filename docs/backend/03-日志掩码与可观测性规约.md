# 03 日志掩码与可观测性规约

> 用途：为了在开发企业级私有化 IM 系统时保障通信内容安全，并满足问题排查（可观测性）的要求，避免大模型大面积将明文消息打印在控制台或文件中。
> 适用范围：Java 后端的所有 Controller, Service, WebSocket Handler。

## 一、日志掩码绝对红线 (Security)

内网 IM 虽然环境收敛，但日志仍禁止明文输出高度敏感数据。以下两类数据绝对**禁止打印**：

### 1.1 `ImMessage` 明文内容禁止打印
无论在日志 `debug` 还是 `info` 级别，**禁止直接使用** `log.info("receive message: {}", message.getContent())`。
*   **正确做法**：日志中只打印消息的元数据：`clientMsgId`、`senderId`、`conversationId`、`messageType`。
*   或者在覆写实体类的 `toString()` 方法时，将 `content` 字段脱敏输出（如：`"content": "***"`）。

### 1.2 `Token` 与用户密码禁止打印
禁止在日志中打印用户的登录凭证和 WebSocket 的认证 Header `Sec-WebSocket-Key` 或 Authorization Bearer 字符串。

## 二、消息可观测性 (Observability)

在大模型生成后端核心链路代码时，必须强制嵌入以下三层核心关注点的日志，使用 `AOP` 优先，如果不使用 `AOP` 也必须在关键代码中主动写入。

### 2.1 HTTP 接口监控 (Controller 层)
在基于 HTTP 的发送接口、历史拉取接口中，需要打印精简的出入参（对 `content` 脱敏）以及**耗时**。

**推荐切面/格式**：
```text
[TraceId-123456] | IM_MSG_API | /api/im/message/send | 耗时: 12ms | 请求方: user_889 | 目标: conv_77
```

### 2.2 WebSocket 长连接全景追踪 (Netty / WS 层)

WebSocket 是 IM 的基石，日志必须覆盖以下完整的长连接生命周期：

| 生命周期阶跃 | 日志级别 | 必打字段 | 说明示例 |
|---|---|---|---|
| **接入认证** | `INFO` | `channelId`, `userId`, `ip` | `[WS-鉴权通过] 用户=1024 上线, 远端IP=192.168.1.5` |
| **异常断开** | `WARN` | `channelId`, `userId`, 异常栈 | `[WS-连接断开] 用户=1024 离线。原因为远程主机强迫关闭一个现有的连接` |
| **主动踢下线** | `INFO` | `userId`, 踢人原因 | `[WS-账号互踢] 用户=1024 在新设备登录，当前 Channel 已关闭` |
| **心跳超时** | `WARN` | `userId`, `idleTime` | `[WS-心跳超时] 用户=1024 超过 60s 未响应 PING，正在强制拆除连接` |
| **消息下发 (Push)**| `DEBUG` | `userId`, `clientMsgId`, `ackType` | `[WS-下发消息] 推送至用户=1024，消息标识=msg-5f3a1...` (禁用 `content`) |

### 2.3 消息重发机制探测日志 (Queue / Retry)

当消息通过 EventBus 或离线重发队列处理时，可能会延时或失败，如果没有任何日志，出现问题将成为黑盒：
*   当系统将失败投递的消息存入离线数据库或暂存队列时，必须打 `INFO` 日志。
*   当系统接收到客户端回推带有 `clientMsgId` 的消息，并命中 `防重校验`（Idempotent Check）而直接抛弃不处理时，必须打 `WARN` 日志说明丢弃原因。

## 三、MDC TraceId 穿透指南 (Java 1.8)

为保证 HTTP 链路生成的 `TraceId` 可以跨越到 WebSocket 或 EventBus 的异步线程池中，必须在线程池提交任务前显式获取转交：

**错误示例（大模型经常犯）**：
```java
eventBus.post(new MessageSendEvent(msg)); // 这里直接丢失 MDC TraceId
```

**正确建议**：
如果项目中没有实现复杂的 MDC 拷贝拦截器，大模型必须在传递的 `Event` 实体或 `Task` `Runnable` 内部构造时，手动塞入 `messageId` 充当 TraceId 的变体角色打印。

## 四、大模型 Prompt 生成约束

当大模型根据需求改写或生成 `*ServiceImpl.java` 或者 `*WebSocketHandler.java` 等文件时：
1. 始终使用 `log.info` 和 `log.warn`，不要使用 `System.out.println`。
2. 看到任何有关于 `content` (正文文本) 或 `fileUrl` (内部绝密文件外链) 的地方，不要带入到日志模板 `"{}"` 中。
3. 如果操作涉及 `clientMsgId`，日志必须包含它以关联全链路。

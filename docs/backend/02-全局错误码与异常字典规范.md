# 02 全局错误码与异常字典规范

> 用途：统一 IM 后端 HTTP 服务与 WebSocket 服务的异常抛出规则，避免大模型生成随意、碎片化的错误代码。
> 适用范围：所有涉及前后端交互的接口响应及长连接断开、消息投递失败处理。

## 一、异常抛出三大红线

1. **绝对禁止吞异常**：严禁出现空的 `catch (Exception e) {}`。
2. **禁止直接返回未包装的原生错误**：所有的 Exception 必须由 `@RestControllerAdvice` 统一拦截，并转化为标准的 JSON 格式返回给前端。
3. **前端状态码必须是 200**：如果是业务异常，HTTP 状态码保持 200，在响应体 JSON 内用 `code` 区分。只有严重的网络/容器级错误（网关宕机、未授权）才允许出现 4xx/5xx HTTP 状态码。

## 二、标准响应结构 (VO)

无论成功失败，所有接口统一使用标准返回体（若依框架的 `AjaxResult` 或 `R`）：

```json
{
  "code": 200,      // 见下方错误码字典
  "msg": "操作成功", // 错误时的前端提示文字
  "data": { ... }   // 成功时携带的具体数据结构
}
```

## 三、错误码字典设计

> 约定：后端通过抛出 `throw new BusinessException(1001, "您已被禁言");` 阻断业务逻辑，前端根据 `code` 渲染交互提示。

### 3.1 基础系统错误码
通用状态码沿用行业规范及若依默认规约。

| Code | 常量名称 | 说明 | 前端处理预期 |
|------|---------|------|-------------|
| 200 | `SUCCESS` | 请求成功 | 取 `data` 渲染 |
| 401 | `UNAUTHORIZED` | 认证失败（Token 过期或非法）| 自动跳转登录页 |
| 403 | `FORBIDDEN` | 无权限操作 | Toast 提示无权限 |
| 500 | `ERROR` | 服务器内部错误 | Toast 提示“系统繁忙”并记录日志 |

### 3.2 消息防重与校验错误码 (10xx)
与 IM 核心消息生命周期相关的异常。

| Code | 常量名称 | 说明 | 前端处理预期 |
|------|---------|------|-------------|
| 1001 | `MSG_SEND_MUTED` | 用户在群内被禁言 / 全员禁言 | 消息气泡变红 `!`，Toast 提示禁言 |
| 1002 | `MSG_NOT_IN_GROUP` | 用户已被移出群组，无法发消息 | 消息气泡变红，隐藏输入框 |
| 1003 | `MSG_SENSITIVE_WORD` | 消息触发敏感词拦截 | 气泡变红，Toast 提示“包含违规内容” |
| 1004 | `MSG_CONTENT_TOO_LONG` | 消息内容超过最大长度限制 | Toast 提示限制长度 |
| 1005 | `MSG_REVOKE_TIMEOUT` | 撤回消息超过 2 分钟时限 | Toast 提示已超时 |
| 1006 | `MSG_IDEMPOTENT_ERROR` | `clientMsgId` 重复，防重拦截 | 不报错，视为成功 |

### 3.3 媒体与文件控制 (20xx)
与上传、文件、图片相关的异常。

| Code | 常量名称 | 说明 | 前端处理预期 |
|------|---------|------|-------------|
| 2001 | `FILE_SIZE_EXCEED` | 文件大小超过单次限制 (100MB) | 阻断并提示超大 |
| 2002 | `FILE_FORMAT_NOT_ALLOWED`| 文件格式在黑名单中 (如 `.exe`) | 阻断并提示禁止上传 |

### 3.4 WebSocket 连接错误码 (4xxx)
用于 WebSocket 握手或断线异常状态码（Standard CloseCodes）。

| CloseCode | 常量名称 | 说明 | 前端处理预期 |
|-----------|---------|------|-------------|
| 4001 | `WS_AUTH_FAILED` | Token 失效切断 | 不重连，退回登录页 |
| 4002 | `WS_KICK_OUT` | 账号在其他设备登录（顶号） | 弹窗提示被踢下线 |
| 4003 | `WS_HEARTBEAT_TIMEOUT` | 心跳超时切断服务器连接 | 前端静默重连 |

## 四、全局异常处理规范

由于是 Java 1.8，必须基于 `@RestControllerAdvice` 和 `@ExceptionHandler` 进行全局拦截：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    private static final Logger log = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    /**
     * 处理自定义业务异常
     */
    @ExceptionHandler(BusinessException.class)
    public AjaxResult processBusinessException(BusinessException e) {
        log.warn("业务异常拦截: [code={}, msg={}]", e.getCode(), e.getMessage());
        return AjaxResult.error(e.getCode(), e.getMessage());
    }

    /**
     * 处理参数校验异常 (JSR-303 @Validated)
     */
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public AjaxResult processBindException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getAllErrors().get(0).getDefaultMessage();
        log.warn("参数校验异常拦截: {}", message);
        return AjaxResult.error(400, message);
    }

    /**
     * 处理未知系统异常
     */
    @ExceptionHandler(Exception.class)
    public AjaxResult processException(Exception e) {
        log.error("系统未知异常拦截:", e);
        return AjaxResult.error(500, "系统繁忙，请稍后再试");
    }
}
```

## 五、大模型生成代码约束 (LLM Prompt)

1. 在编写 Controller 或 Service 层代码时，**绝对不要**用 `try-catch` 将逻辑包装起来并返回一个带有 `"error"` 的 JSON。必须统一 **`throw new BusinessException(...)`** 让全局异常拦截器处理。
2. 对于异常情况，尽量选用以上定义的错误码。如果查无结果，用基础的 `404 (找不到对象)` 或 `400 (参数错误)` 即可。

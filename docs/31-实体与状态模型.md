# 31 实体与状态模型

> 用途：为大模型开发提供稳定的领域事实框架。
> 说明：本文件只写当前已确认的高层模型，不发明未知业务规则。若字段或规则未确认，标记为 `unknown`。

## 1. 核心产品关系

本项目的核心关系是：

1. 用户围绕会话进行沟通
2. 会话承载消息
3. 群组是多人会话的组织载体
4. 文件、搜索、收藏、通话等能力都附着在消息和会话之上

结论：

1. IM 不是一个子模块，而是整个平台的核心领域
2. 其他能力应服务于 IM，而不是与 IM 并列独立演化
3. 当前实现默认建立在“单体 Java 后端 + Vue3 前端 + 内网低并发”前提上
4. 平台整体目标形态参考钉钉 Windows 客户端的模块化办公结构

## 4. 关键状态模型与代码映射

### 4.1 消息发送状态 (MessageStatus)

**Java 枚举定义 (`com.ruoyi.im.constant.MsgStatus`)**:
- `PENDING(0)`: 待发送（进入队列）
- `SENDING(1)`: 发送中（调起 WebSocket）
- `SENT(2)`: 已发送（收到服务端 ACK）
- `DELIVERED(3)`: 已送达（消息已推送到接收方）
- `FAILED(4)`: 发送失败（超时或异常）
- `REVOKED(5)`: 已撤回

**完整状态转换图**:
```text
                    ┌─────────┐
                    │ PENDING │ (待发送)
                    └────┬────┘
                         │ 开始发送
                         ▼
                    ┌─────────┐
          ┌────────│SENDING  │────────┐
          │        └─────────┘        │
          │ 发送成功            发送失败│
          ▼                           ▼
     ┌─────────┐                 ┌─────────┐
     │  SENT   │                 │ FAILED  │◀────┐
     └────┬────┘                 └─────────┘     │
          │                           ▲          │ 重试
          │ 已送达                     │          │
          ▼                           └──────────┘
     ┌───────────┐
     │ DELIVERED │ (已送达)
     └───────────┘
          │
          │ 撤回
          ▼
     ┌─────────┐
     │ REVOKED │ (已撤回)
     └─────────┘
```

**转换触发器**:
| 起始状态 | 目标状态 | 触发事件 | 处理动作 |
| :--- | :--- | :--- | :--- |
| `PENDING` | `SENDING` | 客户端点击发送 | 插入本地 DB，生成 `clientMsgId` |
| `SENDING` | `SENT` | 收到 WS `SEND_ACK` | 更新 `sendStatus=2`, `msgId=server_id` |
| `SENDING` | `FAILED` | 5s 无回执 / 报错 | 更新 `sendStatus=4`, UI 显示重试图标 |
| `SENT` | `DELIVERED` | 收到 WS `DELIVERED_ACK` | 更新 `sendStatus=3`，更新会话未读数 |
| `SENT` | `REVOKED` | 调用撤回接口成功（2分钟内） | 推送 `REVOKE` 帧，前端替换气泡 |
| `FAILED` | `SENDING` | 用户点击重试 | 重新发送，更新 `sendStatus=1` |
| `DELIVERED` | `REVOKED` | 调用撤回接口成功 | 同上，但需同步更新接收方未读数 |

> **重要说明**：
> - `DELIVERED` 状态表示消息已成功推送到接收方设备，用于实现"已送达"功能
> - 撤回操作仅在 `SENT` 或 `DELIVERED` 状态下且发送时间 < 2分钟 时允许
> - 编辑操作仅在 `SENT` 或 `DELIVERED` 状态下且发送时间 < 5分钟 时允许
> - 与 Doc-10 产品需求总纲、Doc-13 需求明细中的状态机定义保持一致

### 8.6 通话状态转换 (Webrtc 信令闭环)

```text
                    ┌─────────┐
                    │  IDLE   │ (空闲)
                    └────┬────┘
                         │ 发起: POST /api/call/initiate
                         ▼
                    ┌──────────┐
          ┌─────────│ INITIATED│ (呼叫中) ────────┐
          │         └──────────┘                 │
          │ 收到: WS[CALL:RINGING]                │ 50s无响应/挂断
          ▼                                      ▼
     ┌─────────┐                            ┌─────────┐
     │ RINGING │ (响铃中)                    │ TIMEOUT │ (超时)
     └────┬────┘                            └────┬────┘
          │                                      │
          ├──── 对方接听: WS[CALL:ANSWER] ───────▶ [CONNECTED] (通话中)
          │                                      │
          └──── 对方拒绝: WS[CALL:REJECT] ───────▶ [REJECTED] (已拒绝)
                                                 │
                                                 ▼
                                            ┌─────────┐
                                            │  ENDED  │ (已结束)
                                            └─────────┘
```

**关键机制说明**:
1. **信令载体**: 通话状态的流转**必须**通过 WebSocket `type:CALL` 帧驱动，严禁仅通过 REST API 轮询。
2. **中间态处理**: 在 `RINGING` 到 `CONNECTED` 之间，前端需保持 `ICE_CANDIDATE` 交换态，此时 UI 应显示"连接中..."。
3. **状态幂等**: 收到重复的 `ANSWER` 或 `REJECT` 信令时，状态机必须保持幂等，不得重复触发挂断逻辑。

## 10. 大模型执行时的使用规则

1. 设计和编码时，优先以"用户 -> 会话 -> 消息"为主轴思考。
2. 文件、收藏、搜索、通话都应回到 IM 主链路验证，不要孤立实现。
3. 若任务会破坏 IM 主链路一致性，应拆小再做。
4. 如果发现实体字段或状态定义不明确，应回写到 [04-待确认问题.md](04-待确认问题.md)。
5. 数据库字段与实体字段映射关系参考本文档第5节。
6. 状态转换逻辑参考本文档第6节的状态转换图。
